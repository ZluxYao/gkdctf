# Q16 - CSRF POST Transfer（Mini Bank）手工复刻

> 靶机：`http://47.120.76.57:34921/`
> 类型：CSRF（同源页面托管 + POST 转账，无 token / 无 Referer 校验）
> 难度：⭐⭐ （新手友好）

---

## 一、为什么这题是 CSRF？

1. 首页提示：
   - 测试账号 `student / student`，余额 10。
   - 有一个 **admin 机器人**，每 20s 用 Chromium 访问 `/pages/<id>`（同源 127.0.0.1）。
   - **admin 向任意用户转账时，第一笔会附带 flag**。
2. 看转账接口：
   ```
   POST /transfer
   to=xxx&amount=1
   ```
   表单页面里写得明明白白 👉 **“无 CSRF token、无 Referer 校验”**。
3. 站点还提供 `/pages/host`：选手可以把任意 HTML 托管为 `/pages/<id>`，**同源**。

把这三块拼起来，思路马上出来：

> 让 admin 机器人去访问我们托管的页面 → 页面用 JS 自动 POST `/transfer` → 浏览器自动带上 admin 的 session cookie → admin 把钱转给我 → flag 进我收件箱。

这就是教科书式的 **CSRF + 同源内容托管**。

---

## 二、四步手工复刻

### Step 1：登录普通账号
浏览器打开 `http://47.120.76.57:34921/login`，用 `student / student` 登录。
登录后导航栏会出现 `/transfer`、`/inbox`、`/pages/host`、`/pages/submit`。

> 观察重点：转账表单里**没有任何隐藏的 token 字段**，说明服务端只靠 Cookie 判定身份。

### Step 2：托管一个“自动提交转账”的页面
进入 `/pages/host`，把下面这段 HTML 贴进文本框，提交：

```html
<!doctype html><html><body>
<form id="f" action="/transfer" method="post">
  <input name="to" value="student">
  <input name="amount" value="1">
</form>
<script>document.getElementById('f').submit();</script>
</body></html>
```

返回类似：`/pages/5a515d38b62a` —— 记下来，这就是我们要喂给 bot 的同源 URL。

> 关键点：表单 `action="/transfer"` 走同源相对路径；JS 自动 `submit()` 触发 POST，浏览器会自动带 cookie。

### Step 3：把页面投递给 admin 机器人
进入 `/pages/submit`，把上一步得到的 `/pages/xxxxxx` 填进去提交。
页面会回显：`已提交，机器人将在 20 秒内访问：/pages/xxxxxx`。

> 注意：bot 要求**必须是 `/pages/<id>`** 形式的同源 URL。这正是为什么我们用 `/pages/host` 托管的原因——跨站页面 bot 不会去访问，而且就算访问也会因为不同源/SameSite 而带不到 admin 的 cookie。

### Step 4：等 20s，去 `/inbox` 收 flag
20 秒内刷新 `/inbox`，会看到：

```
🎁 系统福利发放：admin → student 1 元
本次为 admin 触发的首次福利转账，flag 已一同派送。
🏁 TOGOGO-flag{xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx}
```

搞定。实测得到的 flag：

```
TOGOGO-flag{8460dda9-b26c-4815-b7d6-eac02e064634}
```

## 三、推理链路（题面 → 观察 → 假设 → 验证）

| 步骤 | 观察 | 由此得到的假设 |
| --- | --- | --- |
| 看首页 | 有 admin bot、有“首笔转账带 flag” | 必须借助 admin 身份发起一次转账 |
| 看 `/transfer` | POST 表单、无 token、无 Referer 校验 | 接口是 CSRF 友好的 |
| 看 `/pages/host` | 任意 HTML 托管在**同源** | 可以承载自动提交脚本 |
| 看 `/pages/submit` | bot 只访问 `/pages/<id>` | 限定了交付路径，但同源刚好满足 |

四块拼起来 = CSRF 利用条件齐了。

---

## 四、为什么 POST 不等于安全？

很多新手以为“POST 接口不会被 CSRF”，这是错的。
- 浏览器在跨/同源跳转时会**自动附带 cookie**。
- 只要服务端不校验“请求是不是我自己页面发出的”，就可以伪造。
- 防御标准做法是**任一**：
  1. 不可预测的 **CSRF token**（一次性、绑定 session、双重提交 cookie）。
  2. 严格校验 `Origin` / `Referer`。
  3. Cookie 设 `SameSite=Strict` 或 `Lax`（注意 Lax 仍允许顶级导航 GET）。

本题既无 token、又无 Origin/Referer 校验、又允许同源托管任意 HTML，所以一击必中。

---

## 五、修复建议

1. `/transfer` 必须校验 CSRF token（或 `Origin` 与受信白名单一致）。
2. session cookie 加 `SameSite=Lax/Strict`。
3. `/pages/host` 若必须存在，应禁止内联 `<script>`（用 CSP `script-src 'self'`），或仅作为静态文本展示，避免成为同源 XSS/CSRF 跳板。
