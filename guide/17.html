Q17 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面出现 token/JWT 时，先区分 token 的用途：身份声明、一次性登录令牌、CSRF token 或接口访问凭证。
  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面出现 bot、XSS、CSRF 时，先判断谁会访问页面、访问时携带什么身份、脚本能否同源读取敏感页面。
  • 题面涉及文件包含或 PHP wrapper 时，先确认参数是否进入 include/readfile，再区分源码读取、日志污染和 data wrapper 执行。

建议观察清单

  • 观察 token 的获取接口、提交位置、过期行为和错误响应，再判断能否伪造、复用、侧信道推断或同源读取。
  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用最小化回显或回连验证脚本是否执行，再设计读取 token、代发请求或状态确认链路。
  • 用普通页面名做基线，再测试 wrapper、日志路径或可控请求头，观察响应是否包含源码、执行输出或错误信息。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 令牌类题目要关注签名算法、密钥强度、生命周期、绑定关系和响应时间等旁路信息。
  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • CSRF token 防的是第三方站点伪造请求；如果存在同源 XSS，脚本可能读取页面里的 token 并以受害者身份发起请求。
  • 文件包含题要区分 LFI、RFI、源码泄露和代码执行，修复重点是白名单和禁用危险 wrapper。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q17 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：观察留言和受保护功能

操作

以普通身份登录，分别查看留言板、个人资料和管理员功能，记录哪些内容可提交、哪些页面被拒绝。

预期观察

普通用户可以提交会被管理员查看的内容，但不能直接执行管理员操作。

简要分析

这是典型的“低权限投递内容，高权限用户浏览”的攻击模型。

为什么要这样做

先确认数据流向，才能判断 XSS 是否能借助管理员会话升级影响。

当前步骤验证什么假设

验证用户输入是否可能进入管理员浏览器上下文。

Step 2：验证可控内容是否被当作 HTML 执行

操作

提交无害的 HTML 或脚本探针，查看普通页面或管理员反馈中是否保留标签效果。

预期观察

如果标签未被正确转义，页面行为会与纯文本显示不同。

简要分析

存储型 XSS 的关键是用户输入被保存后在他人浏览器中执行。

为什么要这样做

先用无害探针确认渲染缺陷，避免一开始就构造复杂利用链。

当前步骤验证什么假设

验证留言内容是否存在 HTML/脚本注入能力。

Step 3：用黑盒方式侦察管理员页面要求

操作

设计脚本让管理员浏览器先访问受保护页面并把可见的状态、表单字段或 token 发送到可观察位置。

预期观察

如果 bot 正常访问，可在日志或回显位置看到页面结构线索。

简要分析

复杂 CSRF/XSS 题通常需要先读页面，再按页面要求提交操作。

为什么要这样做

不能假设已经知道管理员表单字段；应让浏览器在真实管理员会话中观察。

当前步骤验证什么假设

验证管理员页面是否包含额外 token 或隐藏字段。

Step 4：构造最终管理员会话操作

操作

根据侦察到的字段，让脚本在管理员浏览器中发起授权或状态改变请求；随后用普通账号检查结果。

预期观察

成功时普通账号状态发生变化，或能访问之前禁止的资料区域。

简要分析

XSS 提供执行环境，CSRF/接口请求完成状态改变，两者结合形成完整链路。

为什么要这样做

把读取字段和提交操作分开，能解释为什么不能直接复制一个固定请求。

当前步骤验证什么假设

验证存储型 XSS 是否能在管理员会话中完成预期操作。

Step 5：记录环境差异和教学结论

操作

如果当前运行中 bot 未触发，把它作为教师复核备注写入证据；学生步骤仍按可验证链路讲解。

预期观察

指引中说明失败时应检查 bot 轮询、等待时间、脚本路径和同源限制。

简要分析

环境问题不应替代漏洞原理，但必须如实记录以便复课时排查。

为什么要这样做

这样既不伪造执行成功，也不让学生只看到参考答案结论。

当前步骤验证什么假设

验证最终材料是否区分了漏洞推理和本次环境执行状态。

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
