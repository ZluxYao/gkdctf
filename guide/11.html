Q11 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面强调验证码时，先判断验证码是否绑定服务端 session、是否一次性失效，以及是否存在前端校验或弱字符集。
  • 题面出现爆破/字典时，先确认账号、密码空间、速率限制、验证码或一次性 token 等约束。

建议观察清单

  • 保持同一个会话访问登录页、获取验证码图片，再提交一次错误密码；如果返回密码错误而不是验证码错误，说明验证码已被服务端接受。
  • 用少量错误样本确认失败响应稳定，再读取题目附件字典或构造有限搜索空间。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 验证码的安全性不只取决于图片难度，还取决于服务端是否正确绑定会话、请求和失败次数。
  • 教学场景中的爆破应强调边界、授权、限速和日志，而不是无约束地扩大尝试范围。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q11 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：黑盒观察登录入口和表单字段

操作

访问 /login.html，确认登录表单的提交地址、用户名字段、密码字段、验证码字段和验证码图片地址。

预期观察

页面返回登录表单，字段为 name、pwd、yzm，验证码图片由 yanzhengma.php 生成。

简要分析

这一步先确认真实请求格式，避免只按题面猜测参数名。

为什么要这样做

验证码题的爆破必须保持同一个 PHP session，否则识别出的验证码和提交请求不属于同一次会话。

当前步骤验证什么假设

验证当前容器确实是 Q11 的验证码登录场景，并确定后续脚本请求所需字段。

Step 2：建立会话并获取验证码图片

操作

用 Python CookieJar 访问登录页，再在同一 opener 中请求 yanzhengma.php 保存验证码图片字节。

预期观察

服务端返回 PNG 验证码，同时 CookieJar 中保留 PHPSESSID。

简要分析

验证码通常绑定 session；先建会话再取图，后续 POST 才能复用这次验证码。

为什么要这样做

如果重新开会话或丢 Cookie，验证码识别正确也会被服务端判错。

当前步骤验证什么假设

验证验证码和登录请求是否依赖同一个 session。

Step 3：OCR 识别验证码并验证可复用

操作

调用 ddddocr 识别 4 位数字验证码，并用一次错误密码请求确认验证码不会立即失效。

预期观察

错误密码时返回“用户名或密码错误”，而不是“验证码不正确”。

简要分析

这说明 OCR 结果被服务端接受，且当前实现允许在同一 session 中复用该验证码继续尝试密码。

为什么要这样做

只有确认验证码通过后，爆破失败响应才代表密码错误；否则会把验证码错误误判为密码错误。

当前步骤验证什么假设

验证题目核心假设：验证码保护退化，不能有效阻止自动化尝试。

Step 4：按题目附件字典枚举账号和密码

操作

读取平台提供的 password2.txt，先尝试题面常见账号，再扩展到实际后台用户名候选，循环 POST login.php。

预期观察

大多数请求返回“用户名或密码错误”；命中时响应中直接出现 flag 弹窗。

简要分析

direct solver 不靠人工观察页面，而是用响应文本中的 flag 模式作为命中条件。

为什么要这样做

教学上要让学生看到：爆破不是盲目发包，而是先固定验证码、固定 session，再对变量进行有边界枚举。

当前步骤验证什么假设

验证字典中是否存在正确口令，以及实际有效用户名是否与题面简写一致。

Step 5：记录结果并生成教学材料

操作

将命中结果写入 flag.txt，把请求证据写入 evidence.md，再生成学生指引 HTML。

预期观察

work/solve/11/flag.txt 存在，work/guide/11.html 是完整 HTML，最终指引中不直接暴露真实 flag。

简要分析

结果文件用于教师备课复核，学生指引则强调观察、推理和验证链路。

为什么要这样做

把复现证据和教学材料分开，既方便教师确认，也避免学生只复制答案。

当前步骤验证什么假设

验证完整流程是否闭环：解题、记录、总结都成功落盘。

执行证据摘要

Q11 direct backend 执行证据

  • 访问登录页并建立 PHP session：HTTP 200
  • OCR 识别 4 位数字验证码，并在同一 session 中复用
  • 使用平台字典 password2.txt 尝试 186 次，命中用户 administrator
  • 命中的密码已用于验证，但最终学生指引中会按占位处理
  • flag 已写入 work/solve/11/flag.txt

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
