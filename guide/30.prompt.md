## 漏洞本质

IDOR（Insecure Direct Object Reference, 不安全的直接对象引用）是 OWASP API 与 Web 安全历年榜首之一（2023 版 OWASP API Top 10 排名 #1 BOLA = Broken Object Level Authorization）。本题是 IDOR 最经典的水平越权（Horizontal Privilege Escalation）子类。

认证（Authentication）vs 授权（Authorization）：

| 维度 | 认证 | 授权 |
| --- | --- | --- |
| 回答的问题 | "你是谁？" | "你能干什么？" |
| 常见实现 | 登录/Session/JWT | 角色/ACL/资源归属 |
| 开发者易错 | 较少（库很成熟） | 较多（业务逻辑层自己写） |
| 本题缺失 | ✅ 做了 | ❌ 没做 |

### 为什么"列表页过滤"救不了你？

开发者写列表页的时候会本能加 `WHERE user_id = ?`——因为"要展示什么"的语义清晰。但详情页的心智是"给我 id=5 的订单"，owner 在这时被忘记了。列表页是"视图过滤"，详情页是"资源访问"，属于两套权限模型，不能混用。

Important

IDOR 的一句话心智模型：

"看得见的（列表过滤）"和"够得着的（详情接口）"必须同步做权限检查，否则拿到任意 id 的人都能读任意资源。

## 关键前置知识

IDOR 四大子类：

| 子类 | 场景 | 举例 |
| --- | --- | --- |
| 水平越权【本题】 | 平级用户之间互读/改资源 | A 用户读 B 用户的订单 / 病历 / 消息 |
| 垂直越权（Q31） | 低权限用户调用高权限接口 | 普通用户访问 `/admin/*` |
| Mass Assignment | 多写字段绕过权限 | 注册时带上 `is_admin=1` |
| BOPLA | 不是对象本身越权，而是对象的某个字段越权 | 只能改 `nickname`，但 `role` 字段没加白名单 |

IDOR 定位常用识别特征：

1. URL 含自增整数： `/order?id=1337`、`/user/42`、`/ticket?no=100023`
2. URL 含 UUID/GUID【并不天然安全，只要在前端或其他接口能枚举就照样中招】
3. URL 含用户名/邮箱： `/profile?user=alice`
4. Cookie 或 Header 带用户标识： `X-User-Id: 42`（改成 43 就成别人）

# 常见坑

| 问题 | 根因 | 解决 |
| --- | --- | --- |
| 登录后再访问 `/order` 仍重定向 login | 没带 cookie | `curl -b /tmp/q30.ck` |
| 浏览器测时 `/order?id=1337` 回登录 | cookie 失效 | 重新登录 |
| 所有 id 都返回样内容 | 缓存或 CDN 命中 | 加随机 query `&_=12345` 破缓存 |
| 猜不到 admin id | 自增序列中 admin 不是固定值 | 扫 `0, 1, 9999, 99999`；或找注册接口批量注册探测规律 |
| UUID 无法枚举 | 无模式 | 去其他接口抓：注册/邀请码/公开分享/消息通知 |
| 403/404 始终返回 | 有 owner 校验或 id 不存在 | 注意区分：`404 对不存在 id` 和 `404 对越权`（后者为正确防护） |
| 返回内容被脱敏 | 服务端返回结果级别做了脱敏 | 检查是否所有字段都脱敏，不同 accept header 可能触发不同响应 |
| 带 PUT/DELETE 的 CSRF 同时存在 | 未校验 owner 也未 CSRF | IDOR + CSRF 叠加，见 Q16/Q17 |

# 关键修复

Important

修一（核心银弹）：owner 校验

[代码块边界]php
$o = $ORDERS[$id] ?? null;
if (!$o || $o['user'] !== $_SESSION['user']) {
  http_response_code(404);   // 返回 404 而非 403，避免枚举
  exit('订单不存在');
}
[代码块边界]

注意返 `404` 不是 `403`：403 等于告诉攻击者“这个 id 存在只是你没权限” → 仍然是信息泄露（Uber 2019 BOLA 就是 403 暴露了行程 id 空间）

修二：数据库层做过滤（最稳）

[代码块边界]sql
SELECT * FROM orders WHERE id = :id AND user_id = :current_user;
[代码块边界]

永远不要先查后判一先把 user_id 写进 WHERE，查不到就直接 404

修三：不可枚举 id（深度防御）

- 把自增整数换成 UUIDv4 / Snowflake
- 但 UUID 不等于安全，仍需配合上面两秀；只能降低爆破效率
- 或用 `hashids` 等不可逆编码（同样仅提高成本）

修四：框架级统一授权

- Laravel: `Policy` / `Gate` / `$this->authorize('view', $order)`
- Spring Security: `@PreAuthorize("@auth.isOwner(#id)")`
- Django: `UserPassesTestMixin`
- 所有资源访问必须经过统一入口，禁止裸 SQL 直接 select

修五：自动化测试

- CI 加"越权测试用例"：用户 A 的 token 去调用用户 B 的资源 id，必须返 404
- 覆盖 GET/POST/PUT/DELETE 全方法

修六：日志与监控

- 记录 `(user_id, resource_owner_id)` 不一致的请求
- 异常频率做告警（一个 session 短时间扫了 N 个不同 id）

修七：语义层抗 BOPLA

- 字段级白名单：更新接口只接受 `['nickname', 'avatar']`，绝不接受 `role/admin`
- 用 DTO/Schema 强制字段过滤
