Q21 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面涉及命令执行时，先确认参数是否拼接到系统命令，再判断是否有回显、过滤字符和可用外带通道。

建议观察清单

  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用基线参数确认正常功能，再用安全的分隔符测试是否改变输出；无回显场景优先找日志或 OOB 记录通道。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • 命令注入的核心是服务端把用户输入当作 shell 语法解释，修复应避免 shell 拼接并使用参数化 API。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q21 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：观察 ping 功能的正常输出

操作

在表单中输入本地回环地址或题目允许的普通主机，提交后记录输出格式。

预期观察

页面显示类似系统 ping 命令的结果。

简要分析

命令注入题通常先表现为一个封装系统工具的 Web 表单。

为什么要这样做

先建立正常输出，后面才能判断异常输入是否改变了系统命令。

当前步骤验证什么假设

验证后端是否调用了系统命令而不是纯库函数。

Step 2：测试异常输入的回显和错误

操作

输入带有无效字符或额外文本的主机名，观察是否出现 shell 错误、命令错误或过滤提示。

预期观察

响应可能包含系统命令错误信息，或仍把输入拼入命令执行。

简要分析

错误信息能暴露命令拼接方式和过滤强度。

为什么要这样做

不应一开始读取敏感文件；先用无害异常输入理解执行环境。

当前步骤验证什么假设

验证用户输入是否未经严格校验就进入命令行。

Step 3：用无害命令验证注入能力

操作

尝试使用命令分隔思路追加一个无害的识别命令，观察响应中是否出现该命令输出。

预期观察

如果响应同时包含 ping 结果和识别命令输出，说明命令注入成立。

简要分析

分隔符让 shell 在同一行执行多个命令；漏洞来自字符串拼接。

为什么要这样做

先验证无害命令能降低误操作风险，也便于课堂演示。

当前步骤验证什么假设

验证是否可以从参数中控制额外 shell 命令执行。

Step 4：定位运行目录和目标文件

操作

通过目录查看或文件存在性探测确认题目结果所在位置，再选择合适的读取方式。

预期观察

响应能显示当前目录、文件列表或目标文件存在性。

简要分析

读取目标前先定位环境，能避免把路径错误误判为漏洞失败。

为什么要这样做

这一步体现从黑盒输出到环境认知的推理过程。

当前步骤验证什么假设

验证目标文件位置和当前用户权限是否满足读取条件。

Step 5：读取题目结果并总结根因

操作

在已验证的注入链上读取目标内容，随后记录修复建议：参数白名单、避免 shell 拼接、使用安全库调用。

预期观察

响应返回题目结果；修复建议能对应到漏洞根因。

简要分析

最终读取只是命令注入验证的闭环。

为什么要这样做

学生应理解根因是 shell 解析了用户输入，而不是 ping 本身危险。

当前步骤验证什么假设

验证完整链路是否从正常功能逐步推导到命令注入。

执行证据摘要

Q21 direct backend 执行证据

  • ping 参数拼接命令注入读取 /flag：HTTP 200
  • flag 已写入 work/solve/21/flag.txt

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
