# Q29 - XXE · XInclude 绕过

实测地址: `http://192.168.1.100:32810/`  
最终 FLAG: `[FLAG 占位]`

## 题目背景

Note

XML Template Merger:

- POST XML 到 `/`，Content-Type `application/xml`
- 页面直白声明: `"为防范 XXE，服务器禁用了 DOCTYPE 与外部实体加载"`
- 服务端行为: `loadXML` → 检测到 `<!DOCTYPE>` 直接 400 → 调 `$doc->xinclude()` → `saveXML()` 回显
- Flag 位于 `/flag`

"Merger [合并]" 一词是灵魂提示——XML 的"合并"官方能力就是 XInclude（W3C 标准）。防护只挡了经典 XXE 三件套（DOCTYPE / ENTITY / entity loader），但 XInclude 不是 entity 机制——这就是整道题的文章。

## 漏洞本质

XXE 防护的"盲区"：开发者心智模型里 `"XXE = DOCTYPE + ENTITY"`，于是防护方案也只针对这两个关键词。 但 XML 规范里还有一条并行赛道——XInclude（XML Inclusions），它：

| 维度 | 经典 XXE | XInclude |
| --- | --- | --- |
| 声明位置 | `<!DOCTYPE>` 内 `<!ENTITY>` | XML 正文里的 `<xi:include>` 节点 |
| 命名空间 | 不需要 | 必须 `xmlns:xi="http://www.w3.org/2001/XInclude"` |
| 触发方式 | parser 默认开启 entity 展开 | 必须显式调用 `$doc->xinclude()` |
| 受哪些开关影响 | `LIBXML_NOENT` / `libxml_disable_entity_loader` / DOCTYPE 黑名单 | 只受 `LIBXML_NOXINCLUDE` 影响（该 flag 很少被用） |
| 本题状态 | ✅ 被三重封死 | ❌ 完全没防 |

\

一句话心智模型：

`XInclude = "XML 版的 #include"`，它不是 entity 机制的"变种"，而是一个独立规范。开发者只要误调了 `xinclude()`，DOCTYPE 怎么拒、entity loader 怎么关，都跟它无关。

\

### 为什么“禁 DOCTYPE + 禁 ENTITY”错觉能防住 XXE？

因为 CTF 和大多数真实 bug 报告里，XXE 攻击的 99% 都走经典 `<!DOCTYPE>[<!ENTITY x SYSTEM "...">]`。于是工程师写防护时只关心这两个字符串——WAF 关键字、parser 禁用选项、代码扫描规则全都围着它们转。XInclude 作为一个"沉默的少数派"，成为盲区。

# 关键前置知识

**XInclude 规范核心：**

[代码块边界]xml
<!-- 形式1：纯文本嵌入（本题用的） -->
<xi:include href="URI" parse="text" encoding="utf-8"/>

<!-- 形式2：嵌入 XML 片段（目标必须是 well-formed XML） -->
<xi:include href="URI" parse="xml"/>

<!-- 形式3：XPointer 精准选择 XML 片段 -->
<xi:include href="URI" xpointer="xpointer(/root/flag[1])"/>

<!-- 形式4：带 fallback，主源失败时替换 -->
<xi:include href="URI" parse="text">
  <xi:fallback>default value</xi:fallback>
</xi:include>
[代码块边界]

`parse="text"` vs `parse="xml"`：

| 属性 | 处理 | 适用 |
| --- | --- | --- |
| `parse="text"` | 把 URI 内容当**纯文本**节点注入 | 读 `flag` / 二进制 / 非 XML 文件（本题） |
| `parse="xml"`（默认） | 当 XML 片段解析，要求 well-formed | 读已知的 XML 模板 |

读 `/flag` 必须 `text`，否则 `[FLAG 占位]` 这种不是合法 XML 的字符串会让 parser 报错。

**各语言对 XInclude 的支持：**

| 语言/库 | 默认是否开 | 关键字 |
| --- | --- | --- |
| PHP DOMDocument | 关，需手动 `$doc->xinclude()` | 本题，极常见考错 |
| Java | `DocumentBuilderFactory.setXIncludeAware(true)` 才开 | Jackson/Spring XML 历史 CVE |
| Python lxml | `etree.parse(..., XMLParser(...)).xinclude()` 手动 | defusedxml 建议禁 |
| .NET XmlDocument | 不原生支持 XInclude | 相对安全 |
| Ruby REXML/Nokogiri | Nokogiri 需 `do_xinclude` 显式调 | REXML 不支持 |

**"XML 家族"里其他容易被忽略的外部加载通道：**

1. `<xsl:import href="URI"/>` — XSLT 导入样式表
2. `<xsl:include href="URI"/>` — XSLT 包含
3. `document('URI')` — XPath / XSLT 函数，读任意 XML
4. `<?xml-stylesheet href="URI"?>` — 处理指令
5. `xml:base="URI"` — 设定基础 URI
6. XInclude — 本题

## 常见坑

| 问题 | 根因 | 解决 |
| --- | --- | --- |
| `<xi:include>` 节点原样返回 | `xinclude()` 没被调 / 命名空间写错 | 核对 ns URI；看服务端是否调了 xinclude |
| 有 `xi:fallback` 的响应里节点被 fallback 替换 | XInclude 跑了但 href 加载失败 | 文件路径不对 / parser 开关把 file 锁了 |
| `libxml_disable_entity_loader(true)` 下 XInclude 也读不到 file | `PHP 7.4` 的联动行为 | 题目设计者的常见误区；修法：去掉该行保留 `LIBXML_NONET` |
| 读 flag 时 parser 报错 | `parse="xml"` 遇到非 well-formed 内容 | 改 `parse="text"` |
| Windows 路径读不到 | `file:///C:/...` 要三斜杠 | `file:///C:/flag.txt` 或 `file://localhost/C:/flag.txt` |
| 读相对路径失败 | `xi:include` 相对路径是相对于 XML 文档 URI；POST body 没文档 URI | 用绝对路径 |
| UTF-8 BOM 干扰 | 文件有 BOM | `encoding="utf-8"` 显式声明，或读 base64 |
| `LIBXML_NONET` 下 http:// 会挂 | 故意的 | 换 `file://`（本题）或 `data://` |
| `data:` URI 可用吗 | 可，但只能静态[无法识别]内容 | `href="data:,hello"` 作测试 |
| CDATA 能藏 XInclude 吗 | 不能，CDATA 里内容是字面量 | 必须让 parser 看到节点 |
