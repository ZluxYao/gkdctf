Q13 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面出现 token/JWT 时，先区分 token 的用途：身份声明、一次性登录令牌、CSRF token 或接口访问凭证。
  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面涉及文件包含或 PHP wrapper 时，先确认参数是否进入 include/readfile，再区分源码读取、日志污染和 data wrapper 执行。
  • 题面涉及命令执行时，先确认参数是否拼接到系统命令，再判断是否有回显、过滤字符和可用外带通道。
  • 题面涉及文件读取或路径参数时，先确认根目录限制，再逐层尝试相对路径，判断是否能越过预期目录。
  • 题面出现爆破/字典时，先确认账号、密码空间、速率限制、验证码或一次性 token 等约束。

建议观察清单

  • 观察 token 的获取接口、提交位置、过期行为和错误响应，再判断能否伪造、复用、侧信道推断或同源读取。
  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用普通页面名做基线，再测试 wrapper、日志路径或可控请求头，观察响应是否包含源码、执行输出或错误信息。
  • 用基线参数确认正常功能，再用安全的分隔符测试是否改变输出；无回显场景优先找日志或 OOB 记录通道。
  • 比较正常文件、缺失文件和上级目录文件的响应差异，用错误信息缩小真实路径。
  • 用少量错误样本确认失败响应稳定，再读取题目附件字典或构造有限搜索空间。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 令牌类题目要关注签名算法、密钥强度、生命周期、绑定关系和响应时间等旁路信息。
  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • 文件包含题要区分 LFI、RFI、源码泄露和代码执行，修复重点是白名单和禁用危险 wrapper。
  • 命令注入的核心是服务端把用户输入当作 shell 语法解释，修复应避免 shell 拼接并使用参数化 API。
  • 路径穿越题的关键是规范化路径和访问控制，不能只靠前端限制文件名。
  • 教学场景中的爆破应强调边界、授权、限速和日志，而不是无约束地扩大尝试范围。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q13 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：黑盒观察登录和管理接口

操作

先用普通身份登录，记录登录响应中的认证凭据；再在不带凭据、带普通凭据两种情况下访问管理接口。

预期观察

未认证请求被拒绝；普通凭据能访问普通功能，但管理接口仍被拒绝。

简要分析

这说明题目关键不在页面隐藏，而在服务端如何判断用户身份和角色。

为什么要这样做

先确认认证边界，后面才知道应该分析凭据结构，而不是盲目猜后台路径。

当前步骤验证什么假设

验证管理接口存在角色校验，并确认普通用户凭据不足以直接访问。

Step 2：解析认证凭据的结构

操作

把凭据按常见 JWT 结构分段解码，只阅读 header 和 payload，不修改请求。

预期观察

可以看到算法、用户名或角色等声明；这些内容是编码不是加密。

简要分析

JWT 的前两段可被客户端读取，真正防篡改依赖签名段。

为什么要这样做

学生需要先理解“可读”和“可改后仍有效”是两回事。

当前步骤验证什么假设

验证身份信息是否由 JWT 声明承载，以及是否可能通过签名校验保护。

Step 3：验证签名强度假设

操作

用已知普通用户 token 离线尝试常见弱密钥候选，比较重新计算出的签名是否与原签名一致。

预期观察

如果某个候选能复现签名，说明服务端使用了可猜测的对称签名密钥。

简要分析

HS 系列 JWT 使用同一个密钥签发和验证；密钥弱时，攻击者可以伪造合法 token。

为什么要这样做

先用已知 token 验证密钥，而不是直接伪造管理员 token，能避免把失败原因混在一起。

当前步骤验证什么假设

验证题目漏洞是否是 JWT 弱密钥，而不是接口本身没有鉴权。

Step 4：构造更高权限声明并重签

操作

只修改与身份或角色有关的声明，使用已验证的候选密钥重新生成签名，再作为认证凭据访问管理接口。

预期观察

服务器接受新凭据后，管理接口返回受保护内容。

简要分析

服务端只要信任 JWT 中的角色声明，伪造签名成功就等价于伪造身份。

为什么要这样做

这一步把前面的弱密钥结论转化为可验证的权限提升。

当前步骤验证什么假设

验证“角色声明可被重签伪造”是否足以通过管理端校验。

Step 5：复盘根因和修复方向

操作

记录完整链路：普通登录、JWT 解码、弱密钥验证、重签访问；教学材料中隐藏真实凭据和最终结果。

预期观察

步骤能解释每次请求为什么发生，而不是只给最终 token。

简要分析

根因是弱签名密钥和过度信任客户端声明。

为什么要这样做

学生应掌握 JWT 的安全边界，避免把 base64 编码误认为安全机制。

当前步骤验证什么假设

验证最终指引是否保留推理过程，并能支持教师讲解修复方案。

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
