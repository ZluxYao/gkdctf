## 漏洞本质

JWT `alg=none` 攻击是 OWASP API Security Top 10 2023 里 `API2: Broken Authentication` 的标志性子类，CVE 级别的历史漏洞：`CVE-2015-9235 / CVE-2018-0114 / CVE-2020-28042` 全属此族。

本题代码骨架（`src/app.py#L11-L18`）：

[代码块边界]python
def decode_token(token: str):
    """漏洞：接受 headers 里声明的算法（None / HS256 都接受）"""
    header = jwt.get_unverified_header(token)   # 未验证的 header 读 alg
    alg = header.get('alg', 'HS256')
    if alg.lower() == 'none':                   # 接受 none 算法
        return jwt.decode(token, options={'verify_signature': False}, algorithms=['none'])
    return jwt.decode(token, JWT_SECRET, algorithms=[alg])

3 段式结构：`header` 公开可读，攻击者可修改 `alg` 字段。**如果服务端信任 header 声明的 alg，就产生本漏洞。**

\

「alg: none」攻击链：

[代码块边界]text
1. 拿一个合法 token（登录 guest 即可）
2. 替换 header: {"alg":"HS256", ...} → {"alg":"none","typ":"JWT"}
3. 替换 payload: {"role":"user"} → {"role":"admin"}
4. 签名段：保留空（none 算法无签名）
   结果：<header-b64>.<payload-b64>.
          ^ 注意末尾保留一个点，签名段为空
5. 服务端读 header.alg=none → 跳过签名验证 → 信任 payload
6. payload.role="admin" → 返回 flag
[代码块边界]

\

为什么 `alg=none` 曾是 JWT 标准的一部分：

- RFC 7519（JWT）+ RFC 7518（JWA）里 `none` 是合法算法名
- 设计初衷：用于**集成测试 / 已经通过其他手段保证完整性**的场景（如 TLS 双向证书）
- 实际后果：大量开发者误用 / 库默认支持 / 导致**签名验证被绕过**

Important

本题一句话心智模型：

“验证 JWT 必须强制指定算法白名单，不能信任 token 自声明的 alg” 否则攻击者只要把 alg 改成 none 就能伪造任意 payload。

## 本题为什么仍能打：

开发者显式在 `alg=='none'` 时走 `options={'verify_signature': False}` 分支，绕过了库的默认防御。这是“反向配置”错误，比老版本库的默认行为更糟。

## JWT 解码的正确姿势：

## 常见坑

| 问题 | 根因 | 解决 |
| --- | --- | --- |
| token 结尾没点被拒 | JWT 格式要求 3 段式 | 保留 `header.payload.` 末尾点 |
| Algorithm not allowed | PyJWT 默认白名单不含 none | 需要服务端显式开 none 分支（本题开了） |
| Signature verification failed | 服务端 algorithms 固定 HS256 未走 none 分支 | 确认服务端代码逻辑 |
| 构造 alg=NONE 大写被拒 | 库区分大小写 | 本题代码是 `alg.lower() == 'none'`，所以大小写都行 |
| payload 里 role=admin 仍被拒 | 服务端可能校验 user=admin | 本题只看 role |
| jwt.io 粘贴无签名 token 报错 | jwt.io 默认期望有签名 | 忽略它，直接 curl 测 |
| 伪造 token 过期被拒 | payload 含 `exp`，已过期 | 加 `exp: 9999999999` 或删掉 `exp` |
