"POP 链"（Property Oriented Programming）术语类比 ROP（Return-Oriented Programming）：

- ROP: 把二进制里现成的 "gadget"（末尾 `ret` 的指令片段）串起来绕过 DEP
- POP: 把代码库里现成的类（含魔术方法）串起来完成任意操作
- 二者共同本质：不注入新代码，复用已有代码

PHP 所有魔术方法（magic methods）都是 POP 跳板：

| 魔术方法 | 触发时机 | 典型危险操作 |
| --- | --- | --- |
| `__construct()` | 对象创建 | 很少是 gadget（需要 `new`） |
| `__destruct()` | 对象销毁 / 脚本结束 | 本题用，最常见 |
| `__wakeup()` | unserialize 时立即触发 | 早期 Yii、Laravel 漏洞 |
| `__toString()` | 对象当字符串使用 | `echo $obj` / 字符串拼接 |
| `__invoke()` | 对象当函数调用 | `$obj()` |
| `__call()` | 调不存在的方法 | `$obj->unknown()` |
| `__get()` / `__set()` | 读/写不存在的属性 | `$obj->x` 触发 |
| `__unserialize()` | PHP 7.4+ 新接口 | 替代 `__wakeup` |

魔术方法互相嵌套 → 可以构造任意长的 POP 链。例如 PHPGGC 里的 Laravel 链：

[代码块边界]text
1  __destruct → __toString → __get → __call → ... → system()
[代码块边界]

## 常见坑

| 问题 | 根因 | 解决 |
| --- | --- | --- |
| `s:5:"/flag"` 长度写错 | PHP serialize 是 字节数 不是字符数 | 多字节（如中文、emoji）要数 UTF-8 字节 |
| protected/private 属性序列化后复制不对 | `\x00*\x00 / \x00Cls\x00` 前缀 | 用 PHP 亲自 serialize 生成，不手写 |
| cookie 含 `+ / = /` 被 URL 解码吞 | base64 含特殊字符 | 用 `base64 -w0` 保证单行；必要时 urlencode |
| `@unserialize` 吞异常，无响应 | 语法错了就全失败 | 先本地 `php -r 'var_dump(unserialize("..."))'` 验证 |
| enabled=b:0 -> 无输出 | 代码里 `if ($this->enabled) {}` | 改 `b:1` |
| `is_file('/flag')` 返回 false | 文件不存在 / 权限问题 | 先 `is_file('/etc/passwd')` 探测 |
| `__destruct` 不触发 | PHP 7.0.15+ 对 unserialize 失败的对象不调 `__destruct` | 确保 serialize 语法正确 |
| `O:10:"LogCleaner":2:{...}` 属性数写 3 但只 2 个 | 会导致 unserialize 失败 | 严格对齐 |
| PHP 7.0.10+ 的 `__wakeup` 绕过被修 | 需要更老版本 | 本题 PHP 7.4 已修：改走 `__destruct` |

## 关键修复

Important

修一（核心银弹）：完全不用 unserialize 处理用户输入

[代码块边界]php
// 危险：unserialize 用户可控数据
$pref = @unserialize(base64_decode($_COOKIE['pref']));

// 方案 A：改用 JSON（不会触发类构造）
$pref = json_decode($_COOKIE['pref'], true);
$theme = $pref['theme'] ?? 'light';
$lang  = $pref['lang']  ?? 'zh';

// 方案 B：allowed_classes 白名单（PHP 7.0+）
$pref = unserialize(base64_decode($_COOKIE['pref']), [
  'allowed_classes' => ['UserPref']  // 只允许 UserPref
]);
// 传 false 完全禁止 -> 其他类会变 __PHP_Incomplete_Class

[代码块边界]php
function sign($data) {
  global $SECRET_KEY;
  $mac = hash_hmac('sha256', $data, $SECRET_KEY);
  return base64_encode($data) . '.' . $mac;
}

function verify_unsign($sig) {
  global $SECRET_KEY;
  [$b64, $mac] = explode('.', $sig, 2);
  $data = base64_decode($b64, true);
  $expected = hash_hmac('sha256', $data, $SECRET_KEY);
  if (!hash_equals($expected, $mac)) throw new Exception('tampered');
  return $data;
}

// Cookie: pref=<base64>.<hmac>
// 篡改即破坏签名 -> 拒绝
[代码块边界]

### 修六: PHP 版本升级

- PHP 7.0.10+ / 7.1.0+ 修了 `__wakeup` 绕过（CVE-2016-7124）
- PHP 7.4+ 引入 `__serialize` / `__unserialize` 更安全的接口
- 最新版本永远还是最好的防御

### 修七: 容器/最小权限

[代码块边界]dockerfile
# Apache 跑 www-data 而非 root

# /flag 权限 640 root:root -> www-data 读不到
RUN (chown root:root /flag && chmod 640 /flag
# （但本题 www-data 能读 /flag，因此靶机 flag 权限设成 644）
[代码块边界]

### 修八: 监控与告警

- WAF 拦截 request body / cookie 里含 `O:N:` 的请求（PHP 序列化指纹）
- `php-fpm.log` 监控 `__destruct` 异常 stack trace
- auditd 监控 `/flag` / `/etc/shadow` 等敏感文件的 open syscall
