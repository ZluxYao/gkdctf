## 漏洞本质

LFI（Local File Inclusion）核心：`include` / `require` 接用户可控文件名。具体到本题：

[代码块边界]php
$page = $_GET['page'] ?? 'home';    // 无黑名单、无白名单
$file = $page . '.php';             // 只拼后缀
@include $file;                     // 直接包含
[代码块边界]

本题特殊性：flag 在源码注释中——不能直接 `?page=index`，原因：

- `?page=index` → `include 'index.php'` → PHP 会解析而不是输出
- PHP 解释器看到 `<?php ... ?>` 会执行代码，注释会被吃掉
- 页面最多可见 HTML 输出，注释永远看不到

因此需要一种方法：让 PHP 把 `index.php` 当成“数据”而不是“代码”读出来——这正是 `php://filter` 的用武之地。

> Important
>
> 伪协议 × LFI 的四种角色：

| 目的 | 伪协议 | 关键技术 |
|---|---|---|
| 读源码（本题） | `php://filter/convert.base64-encode/resource=…` | base64 让代码降格为数据 |


## Page 71

| 目的 | 伪协议 | 关键技术 |
| --- | --- | --- |
| 读任意文件 | `/etc/passwd`、`file://…` | 直接放进 `include` |
| RCE（执行命令） | `data://text/plain,<?php system($_GET[c]);?>` | `allow_url_include=On` |
| RCE·伪影进入 | `phar://upload/pic.jpg` | 带反序列化的 `phar` |

# 关键前置知识

PHP 流包装器（Stream Wrapper）天幕：

| Wrapper | 语义 | 举例 |
| --- | --- | --- |
| `file://` | 本地文件（默认） | `file:///etc/passwd` |
| `php://filter` | 在读写时套过滤器 | `php://filter/read=…/resource=…` |
| `php://input` | 读请求体 | POST 时代 `include` 读进 RCE |
| `data://` | URL 形内联数据 | `data://text/plain;base64,PD9waHA…` |
| `zip://` / `phar://` | 压缩包内部文件 | `phar://x.jpg/[敏感路径占位]` |
| `expect://` | 执行命令 | `expect://ls`（需 expect 扩展） |

`php://filter` 全形语法：

[代码块边界]text
php://filter/[read=]<过滤器链>/[write=]<过滤器链>/resource=<源路径>
[代码块边界]

过滤器入营四大族（番古碰计）：

| 族 | 样子 | 角色 |
| --- | --- | --- |
| 字符串过滤器 | `string.rot13`、`string.toupper` | 简单变换 |
| 转码过滤器 | `convert.base64-encode`、`convert.base64-decode` | 本题主力——让代码变为“数据” |
| 压缩过滤器 | `zlib.deflate`、`zlib.inflate` | gzip 压缩 |
| 加密过滤器 | `mcrypt.*`、`mdecrypt.*` | PHP 7.2 后已废 |

为什么 `base64-encode` 是牙“银弹”：

- PHP 解析 `<?php ... ?>` 是在“读完文件后”进行的
- `filter` 发生在“读的当口”，每读一块数据就应用一次 base64 编码
- 编码后的字符串 不再包含 `<?php`，解析器看到 `PD9waHA…` 以为是普通文本只能写到输出

> Caution
>
> `is_file()` / `file_exists()` 对 `php://filter` 伪协议返回 `false`。若题目在 `include` 前做 `is_file` 会把伪协议试托靠马一本题异果原版有此点杀龟，已在第一轮纪演发现并变作 `@include $file` 半开。

# 解题思路推导

## 阶段 ①：刚到站手一挤注意 `?page=`

首页上快签指台菜单 `/?page=home`、`/?page=about`，想都不用想：后端肯定是直接换编 `$page . ".php"` 等货色。这个模式是经典 PHP LFI 的指标。

例饭验证：请求 `?page=home` → 回显 `home.php` 内容 ✓。换个不存在的 `?page=foo`：回“页面不存在”（或空）→ 作我给了 `include`。

## 阶段 ②：先者 `?page=/etc/passwd`——符号 `.php` 干扰

标准第一抽块：

悲观结论：普通文件读不了，但别瞎蒙——题目的 flag 本就在 PHP 文件里。

## 阶段 ③：目标确定为 index.php，且必须以“数据”样读出

`.php` 拼接不恶意下，目标文件名尾已为 `.php`，正好 → 直接 `?page=index` 将 `$file = "index.php"`。但 PHP 会把它交解析器，注释看不见。

因此 → `php://filter/convert.base64-encode/resource=index` ——这是源码读取的值义黑美峰。

# php://filter 语法拆解

[代码块边界]text
1  php://filter / convert.base64-encode / resource = index
2         │                 │                            │
3         │                 └── 编码方式（将他转为数据流）  └── 目标资源（由后端拼 .php 补全）
4         │
5         └── 流包装器名
[代码块边界]

**多重过滤器链（第二展开形，可裙控 WAF 对 filter 关键字的黑名单）：**

[代码块边界]text
1  php://filter/read=string.rot13|convert.base64-encode/resource=index
2            │          │                    │
3            │          └── 去 ── 管道依次应用
4            └── 指定 read 或 write，默认可省

**其他常见的 base64 替代方案（过滤关键字时）：**

- `convert.iconv.UTF-8.UTF-7` → 转 UTF-7 做编码

- `zlib.deflate` → gzip 压缩，再 base64 容易输出

# 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| filter 回 `false` 或空 | 题目用 `is_file` / `file_exists` 检查 → 伪协议不算文件 | 换题目或换思路你的通街 |
| base64 段几行混在 HTML 中 | PHP 会把 HTML 顶贴先输出 | `grep -oE '[A-Za-z0-9+/=]{100,}'` 过滤、取最长块 |
| `resource=index.php` vs `resource=index` | 后端拼 `.php`，写 `index.php` 会变 `index.php.php` | 看理点逻辑，本题写 `index` 正好 |
| `allow_url_include=Off` 无法用 `data://` | 含有 URL 的伪协议被限 | `php://filter` 不受此限（本地流） |
| 多轮编码但输出亦类效 | 开了 `open_basedir` | 换目标在 basedir 内的文件 |
| PHP 7.4 `%00` 截断截不动 | PHP 5.3.4+ 已修复 | 换思路一用 filter |

# LFI 进阶谱（从读源码往 RCE）

> Important

1. 读源码（本题）：`php://filter/convert.base64-encode/resource=...`
2. 读日志文件（Q19）：`?page=/var/log/apache2/access.log` 配合 UA 注入就是 RCE
3. `data://` 直挂代码：`?page=data://text/plain,<?=phpinfo();?>`（需 `allow_url_include=On`）
4. `php://input` + POST：`?page=php://input` + POST body 为 PHP 代码
5. `phar://` 反序列化：配合一个上传点上传 phar，`?page=phar://x.jpg/test` 触发 `__destruct`
6. `session.upload_progress`：往 PHP session 里注一段代码后 include，无上传点也能 RCE
7. SSH `auth.log`：用恶意用户名做 SSH 登录，注入 PHP 到 `/var/log/auth.log`
8. `/proc/self/environ`（旧 PHP）：控制 UA、Cookie 于环境变量 → include 当代码跑

# 关键修复

> Important

**修一（核心）：白名单 + `basename()` — 根绝伪协议与路径穿越**

[代码块边界]php
$whitelist = ['home','about','contact'];
$page = basename($_GET['page'] ?? 'home');
if (!in_array($page, $whitelist, true)) { $page = 'home'; }
include __DIR__ . '/' . $page . '.php';
[代码块边界]

- `basename` 打掉任何路径分隔符、伪协议前缀
- `in_array` 白名单 → 代码载入路径完全已知
- 用 `__DIR__` 防相对路径被游成

**修二：PHP 全局开关双防**

[代码块边界]ini
; php.ini
allow_url_include = Off        ; 杀 data://、http://
allow_url_fopen   = Off        ; 不需要读远程就关
open_basedir      = /var/www/html/  ; 锁在你的目录里
[代码块边界]

- `allow_url_include=Off` 和 `allow_url_fopen=Off` 到 90% 的 include扛 LFI
- `open_basedir` 让 `/etc/passwd` 和日志文件也读不了

**修三：对 include 参数一律黑名单特殊字符**

[代码块边界]php
if (preg_match('#[:/\\\\\0]#', $page)) { http_response_code(400); exit('bad'); }
// 屏蔽 “:” (伪协议冒号)、/、\、\0
[代码块边界]

**修四：模板引擎隔离—别把模板名给用户来拼**

- Twig / Smarty 编译模板 → include 的不再是原生 PHP 文件与用户输入链接

**修五：敏感文件放到 webroot 之外**

[代码块边界]text
/var/app/secret/[敏感路径占位]    → 放配置 / flag
/var/www/html/                → 只放对外模板
[代码块边界]

即使 LFI 让路径，open_basedir 也撬上心火。
