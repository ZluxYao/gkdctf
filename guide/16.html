Q16 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面出现 token/JWT 时，先区分 token 的用途：身份声明、一次性登录令牌、CSRF token 或接口访问凭证。
  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面出现 bot、XSS、CSRF 时，先判断谁会访问页面、访问时携带什么身份、脚本能否同源读取敏感页面。
  • 题面涉及文件包含或 PHP wrapper 时，先确认参数是否进入 include/readfile，再区分源码读取、日志污染和 data wrapper 执行。
  • 题面涉及文件读取或路径参数时，先确认根目录限制，再逐层尝试相对路径，判断是否能越过预期目录。

建议观察清单

  • 观察 token 的获取接口、提交位置、过期行为和错误响应，再判断能否伪造、复用、侧信道推断或同源读取。
  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用最小化回显或回连验证脚本是否执行，再设计读取 token、代发请求或状态确认链路。
  • 用普通页面名做基线，再测试 wrapper、日志路径或可控请求头，观察响应是否包含源码、执行输出或错误信息。
  • 比较正常文件、缺失文件和上级目录文件的响应差异，用错误信息缩小真实路径。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 令牌类题目要关注签名算法、密钥强度、生命周期、绑定关系和响应时间等旁路信息。
  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • CSRF token 防的是第三方站点伪造请求；如果存在同源 XSS，脚本可能读取页面里的 token 并以受害者身份发起请求。
  • 文件包含题要区分 LFI、RFI、源码泄露和代码执行，修复重点是白名单和禁用危险 wrapper。
  • 路径穿越题的关键是规范化路径和访问控制，不能只靠前端限制文件名。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q16 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：观察转账和管理员访问通道

操作

登录普通账号，查看转账表单、页面托管功能和提交给管理员访问的入口。

预期观察

转账需要登录态，题目还允许提交一个站内页面供管理员访问。

简要分析

这说明攻击面可能是同源页面承载自动提交表单。

为什么要这样做

先区分普通跨站请求和站内托管页面，才能解释为什么浏览器会带管理员 Cookie。

当前步骤验证什么假设

验证是否存在可让管理员访问自定义内容的同源位置。

Step 2：建立转账请求基线

操作

在自己账号上提交一次无害转账或测试请求，记录请求方法、字段名、成功和失败响应。

预期观察

服务端接受 POST 表单，但页面中看不到不可预测的 CSRF token。

简要分析

如果没有 CSRF token，攻击者可构造同样字段的表单让受害者自动提交。

为什么要这样做

先记录字段名和方法，后续构造页面才不会因为参数错误失败。

当前步骤验证什么假设

验证转账接口是否只依赖 Cookie 鉴权，而缺少请求来源证明。

Step 3：托管自动提交页面

操作

在题目提供的页面托管处放置一个只包含表单和自动提交逻辑的页面，目标指向转账接口。

预期观察

页面被访问时，浏览器会自动向同源接口发起 POST 请求。

简要分析

站内托管让请求处于同源环境，很多简单 SameSite 防护无法阻止。

为什么要这样做

优先用脚本复现生成页面，同时在教学中说明界面如何填写托管内容。

当前步骤验证什么假设

验证自定义页面能否代替用户点击提交转账表单。

Step 4：提交给管理员并检查结果

操作

把托管页面地址提交给管理员访问队列，等待处理后查看自己的消息、余额或题目指定结果页。

预期观察

成功时可观察到管理员会话执行了预期状态改变，并返回题目结果。

简要分析

CSRF 成功的证据应来自状态变化或结果页，而不是假设 bot 一定执行。

为什么要这样做

等待后检查结果能区分页面构造失败、bot 未访问和接口拒绝三种情况。

当前步骤验证什么假设

验证管理员浏览器是否在高权限会话下提交了表单。

Step 5：解释防护策略

操作

记录修复建议：每个敏感 POST 加不可预测 CSRF token，校验 Origin，并避免用户可控同源页面承载任意脚本。

预期观察

学生能理解 POST 本身并不等于防 CSRF。

简要分析

本题核心是浏览器自动带 Cookie 与服务端缺少请求意图校验。

为什么要这样做

修复策略应覆盖接口和内容托管两个层面。

当前步骤验证什么假设

验证教学步骤是否从请求基线推导到同源自动提交。

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
