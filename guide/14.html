Q14 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面出现 token/JWT 时，先区分 token 的用途：身份声明、一次性登录令牌、CSRF token 或接口访问凭证。
  • 题面出现 bot、XSS、CSRF 时，先判断谁会访问页面、访问时携带什么身份、脚本能否同源读取敏感页面。
  • 题面涉及文件读取或路径参数时，先确认根目录限制，再逐层尝试相对路径，判断是否能越过预期目录。
  • 题面出现爆破/字典时，先确认账号、密码空间、速率限制、验证码或一次性 token 等约束。

建议观察清单

  • 观察 token 的获取接口、提交位置、过期行为和错误响应，再判断能否伪造、复用、侧信道推断或同源读取。
  • 用最小化回显或回连验证脚本是否执行，再设计读取 token、代发请求或状态确认链路。
  • 比较正常文件、缺失文件和上级目录文件的响应差异，用错误信息缩小真实路径。
  • 用少量错误样本确认失败响应稳定，再读取题目附件字典或构造有限搜索空间。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 令牌类题目要关注签名算法、密钥强度、生命周期、绑定关系和响应时间等旁路信息。
  • CSRF token 防的是第三方站点伪造请求；如果存在同源 XSS，脚本可能读取页面里的 token 并以受害者身份发起请求。
  • 路径穿越题的关键是规范化路径和访问控制，不能只靠前端限制文件名。
  • 教学场景中的爆破应强调边界、授权、限速和日志，而不是无约束地扩大尝试范围。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q14 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：观察登录流程和一次性 token

操作

访问登录页或接口，记录是否需要先获取 token；用错误账号密码尝试一次完整登录。

预期观察

服务端要求携带 token，且每次尝试都需要新 token 或会改变 token 状态。

简要分析

这说明自动化验证必须把“取 token”和“提交登录”作为一个原子流程。

为什么要这样做

先搞清楚请求依赖，避免后面时序测试因为 token 失效而误判。

当前步骤验证什么假设

验证登录失败是否可重复测量，以及 token 是否影响每次请求。

Step 2：建立错误密码耗时基线

操作

固定用户名，使用同长度但明显错误的密码，多次获取新 token 并提交，记录响应耗时和失败提示。

预期观察

失败提示一致，但不同输入的耗时可能存在可测量差异。

简要分析

如果服务端逐字符比较并在第一个错误处提前返回，正确前缀越长耗时越长。

为什么要这样做

先测基线能知道网络抖动范围，避免把偶然慢请求当作正确字符。

当前步骤验证什么假设

验证登录接口是否存在可用于区分前缀正确性的时序侧信道。

Step 3：逐位推断候选口令

操作

从空前缀开始，枚举下一位候选字符；每个候选重复多次测量，选择稳定更慢的候选作为当前前缀。

预期观察

某些候选在多轮采样中更稳定地接近最长耗时。

简要分析

这不是猜答案，而是用响应时间把搜索空间逐步缩小。

为什么要这样做

逐位推进可以把指数级尝试转化为每一位的有限枚举。

当前步骤验证什么假设

验证当前前缀是否更可能匹配真实口令前缀。

Step 4：用推断口令做最终登录验证

操作

获取新的 token，使用完整推断口令提交登录；如果失败，回退到噪声较大的位置增加采样。

预期观察

成功时页面或接口返回受保护内容；失败时仍保持普通错误提示。

简要分析

最终登录请求是对时序推断链的闭环验证。

为什么要这样做

时序攻击会受网络波动影响，必须允许回退和重采样。

当前步骤验证什么假设

验证推断出的口令是否真实有效，而不只是统计上看起来可疑。

Step 5：说明安全根因

操作

在教学记录中说明应使用恒定时间比较、统一错误路径和限速策略。

预期观察

学生能区分“弱口令爆破”和“时序侧信道”两类问题。

简要分析

本题的重点是比较函数泄露了部分正确性，而不是题面直接给出密码。

为什么要这样做

把修复方向写清楚，能帮助学生理解为什么简单改提示语并不充分。

当前步骤验证什么假设

验证学生指引是否解释了 token、采样和侧信道三者关系。

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
