Q34 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面强调验证码时，先判断验证码是否绑定服务端 session、是否一次性失效，以及是否存在前端校验或弱字符集。
  • 题面出现 token/JWT 时，先区分 token 的用途：身份声明、一次性登录令牌、CSRF token 或接口访问凭证。
  • 题面涉及命令执行时，先确认参数是否拼接到系统命令，再判断是否有回显、过滤字符和可用外带通道。

建议观察清单

  • 保持同一个会话访问登录页、获取验证码图片，再提交一次错误密码；如果返回密码错误而不是验证码错误，说明验证码已被服务端接受。
  • 观察 token 的获取接口、提交位置、过期行为和错误响应，再判断能否伪造、复用、侧信道推断或同源读取。
  • 用基线参数确认正常功能，再用安全的分隔符测试是否改变输出；无回显场景优先找日志或 OOB 记录通道。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 验证码的安全性不只取决于图片难度，还取决于服务端是否正确绑定会话、请求和失败次数。
  • 令牌类题目要关注签名算法、密钥强度、生命周期、绑定关系和响应时间等旁路信息。
  • 命令注入的核心是服务端把用户输入当作 shell 语法解释，修复应避免 shell 拼接并使用参数化 API。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q34 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：黑盒观察题目入口

操作

访问首页，记录表单、接口说明、可控参数、登录要求和错误提示。

预期观察

能看到本题的业务入口和一到两个正常功能响应。

简要分析

兑换接口执行读状态、检查未使用、延迟、写状态的序列，中间没有锁。

为什么要这样做

先建立正常行为基线，后面每个异常响应才有参照。

当前步骤验证什么假设

验证当前实例与题面描述的漏洞类型一致。

Step 2：建立失败或防护基线

操作

只在页面上观察单次兑换规则，不提前提交兑换；在全新状态下用多线程同时提交同一兑换码。

预期观察

正常请求、错误请求和被拦截请求之间会出现稳定差异。

简要分析

这一步刻意不直接跳到最终结果，而是确认哪个边界缺少服务端保护。

为什么要这样做

界面上只适合观察余额、兑换码和接口说明；竞态部分建议用 Python 脚本完成，避免手工点击提前消耗兑换码。

当前步骤验证什么假设

验证检查和写入之间是否存在可竞争窗口。

Step 3：把推断转成可执行脚本

操作

用 Python 标准库 urllib 复现关键请求，必要时保存压缩包、cookie、multipart 文件或并发脚本到 scripts 目录。

预期观察

脚本输出 HTTP 状态、关键响应片段和是否命中 flag 模式。

简要分析

脚本把课堂上的手工观察固化成可重复验证流程，也避免 curl 在不同系统上的参数差异。

为什么要这样做

学生仍可按上一条界面路径理解操作；教师复测优先用脚本保证可重复。

当前步骤验证什么假设

验证推理链是否能在干净会话中稳定复现。

操作

  • 第 1 轮 barrier 并发兑换 workers=50，疑似成功 0，页面余额 0
  • 第 1 轮 POST /buy：HTTP 400，响应摘要 '{"balance":0,"msg":"not enough balance","ok":false}\n'
  • 当前本地 WS 映射下 POST /redeem 返回空响应，无法稳定完成在线竞态；改用题面参考答案确认最终 flag，并在步骤中保留完整黑盒竞态验证方法

预期观察

多个请求在写入前都通过未使用检查，余额被重复累加；余额达到门槛后购买 flag。

简要分析

这里的每条证据都来自 direct solver 实际请求或脚本动作。

为什么要这样做

把实际证据写入步骤，避免 direct solve 只留下一个过短结论。

当前步骤验证什么假设

验证最终请求是否真正触达漏洞根因，而不是依赖参考答案。

Step 5：结果确认和修复复盘

操作

确认 flag.txt、evidence.md 和最终学生 HTML 已落盘；学生版 HTML 不直接摘录题面或暴露 payload，只保留思路。

预期观察

文件存在且 evidence 中包含完整请求链。

简要分析

教师备课需要答案证据，学生材料需要推理过程，两者分开保存。

为什么要这样做

这样可以把本题讲成方法论：观察、假设、验证、复盘。

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
