## 漏洞本质

垂直越权（Vertical Privilege Escalation）/ BFLA（Broken Function Level Authorization），OWASP API Top 10 2023 位列 #5，是企业级应用里数量最多、最低级、最难治理的权限漏洞之一。

与水平越权（Q30）的区别：

| 维度 | 水平越权（Q30） | 垂直越权（本题） |
| --- | --- | --- |
| 攻击对象 | 同一角色的不同用户的资源 | 低权限用户调用高权限接口 |
| 攻击面 | 资源 id 枚举 | 接口路径枚举 |
| 典型漏点 | `"展示"`接口漏 owner 校验 | `"功能"`接口漏 role 校验 |
| 题目中 | A 用户读 B 用户订单 | user 调用只有 admin 能用的功能 |
| OWASP | BOLA（#1） | BFLA（#5） |

为什么这种漏洞数量很多？

1. 入口页最显眼 -> code review 优先看它，检查必然加
2. 子接口散布在多个文件/多个 commit -> 容易漏一个
3. 加新功能时（`/admin/export_v2`）-> 开发者忘了同步复制权限 check
4. 框架层 middleware 漏配（某条路由没挂 `@admin_required`）
5. 隐藏功能（前端 JS 不暴露 `/admin/secret` 的 URL，开发者以为"攻击者不知道这条路径 = 安全"）— 经典 Security through Obscurity 错觉

Important

**本题一句话心智模型：**

“前端不给按钮”不等于“后端做了权限校验”——后端必须对每一个敏感接口独立、显式、统一地校验角色；遗漏任何一条，整个管理面板就破防。

# 关键前置知识

**垂直越权的四大子模式：**

| 模式 | 特征 | 举例 |
| --- | --- | --- |
| 隐藏路径 | 入口不暴露在 UI，但服务端路由存在 | 本题 `/admin/secret` |
| 前端隐藏按钮 | 接口对所有人开放，只是前端不展示 | 普通用户抓包拿到 admin 的 curl |
| Mass Assignment | 注册/更新接口接受 `role` / `is_admin` 字段 | `POST /register {"username":"x","role":"admin"}` |
| 可控角色字段 | JWT / cookie / localStorage 里有 `role` 字段 | 改 JWT payload 的 `role: admin` 再签 |

**垂直越权的经典发现手法：**

1. 子路径字典爆破（本题）：`gobuster` / `dirsearch` / `ffuf`，字典选 `admin-panels.txt`、`raft-small-words.txt`
2. JS 源码挖掘：前端 `app.js` 里偶尔硬编码后台路径
3. `robots.txt` / `sitemap.xml`：管理员路径有时被错误放进
4. `.git` / `.svn` / 备份文件：Q12 讲过
5. HTTP `OPTIONS` / `TRACE`：某些服务器响应里暴露可用方法/路径
6. 接口文档泄露：Swagger/Postman 集合、`/api-docs`、`/swagger-ui`
7. 错误信息泄露：故意触发 500 错误，栈跟踪暴露内部路径

## 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| 所有子路径都 200 | FallbackResource 把 404 改写首页 | 按 size 差异 / 内容指纹过滤 |
| 所有子路径都 302 -> /login | 服务端所有路由都要登录 | 先登录拿 cookie 再爆 |
| 403/200 混着返 | 有些路径做了检查、有些没做 | 专门挑 200 且 size 异常的 |
| 爆破太慢 | 字典太大 | 用 admin-panels 字典(通常 200 条) |
| WAF/Rate limit | 请求过快被封 | -t 5 调并发, 加随机 User-Agent |
| 字典漏掉关键词 | 业务有自己的命名风格 | 看 JS/HTML 正文, 提取关键词自制字典 |
| 未登录也能访问 flag 接口 | 完全无 auth (更严重) | 试一下未登录直接访问，有时直接送 |
| 接口是 POST 才触发 | 默认 GET 返 405 或空 | 试 POST/PUT + 各种 body |

## 垂直越权进阶 7 谱

Important

1. 隐藏子路径（本题）：权限检查漏洞点，靠爆破发现
2. 前端隐藏按钮：所有用户都能调，只是 UI 不展示
3. Mass Assignment：注册/更新接口接受 `role` / `is_admin` 字段
4. JWT Claim 篡改：`role:user` 改成 `role:admin` [配 Q13 / Q35]
5. Header/Cookie 注入：`X-Admin: 1` 等非标准头被后端信任
6. 二阶越权：先用水平越权拿 admin 的资源 id，再发起对该 id 的写操作
7. 框架路由级漏洞：Spring `PathPattern`、Express `use()` 顺序错误致 middleware 不生效

### 修二（框架层）：用 middleware 而不是手写 if

- Laravel: `Route::middleware('role:admin')->group(function () { ... })`
- Spring: `@PreAuthorize("hasRole('ADMIN')")` + `@RequestMapping("/admin/**")`
- Express: `router.use('/admin', requireAdmin)` + 所有 `/admin/*` 自动经过
- Django: `@user_passes_test(is_admin)`

### 修三：编译时/部署时的“权限总览表”生成

- 写一个脚本扫所有路由，自动生成“路径 -> 所需角色”清单
- Code review 必审这张表，任何新加路由必须在表里出现
- CI 检查：路由数量 == 权限表条目数，不等就 fail

### 修四：DAST/SAST 自动化扫权限漏洞

- 低权限 token 跑一遍所有路由，任何 2xx 响应都是风险点
- 每次 CI 跑，不通过就不允许合并

### 修五：Defense in Depth

- 业务逻辑层再做一次权限校验（双保险）
- 数据库层：`WHERE tenant_id = :current` 作为底牌

### 修六：降低攻击面

- 敏感管理路径用随机后缀（`/admin-a7f3b92/secret`）— 仅作为额外防线，不是主力
- Web 服务器层做 IP 白名单（`<Location /admin>Require ip 10.0.0.0/8</Location>`）
- 管理面板单独部署在内网域名/VPN 后面
