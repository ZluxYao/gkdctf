Q19 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面涉及文件包含或 PHP wrapper 时，先确认参数是否进入 include/readfile，再区分源码读取、日志污染和 data wrapper 执行。
  • 题面涉及命令执行时，先确认参数是否拼接到系统命令，再判断是否有回显、过滤字符和可用外带通道。
  • 题面涉及文件读取或路径参数时，先确认根目录限制，再逐层尝试相对路径，判断是否能越过预期目录。

建议观察清单

  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用普通页面名做基线，再测试 wrapper、日志路径或可控请求头，观察响应是否包含源码、执行输出或错误信息。
  • 用基线参数确认正常功能，再用安全的分隔符测试是否改变输出；无回显场景优先找日志或 OOB 记录通道。
  • 比较正常文件、缺失文件和上级目录文件的响应差异，用错误信息缩小真实路径。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • 文件包含题要区分 LFI、RFI、源码泄露和代码执行，修复重点是白名单和禁用危险 wrapper。
  • 命令注入的核心是服务端把用户输入当作 shell 语法解释，修复应避免 shell 拼接并使用参数化 API。
  • 路径穿越题的关键是规范化路径和访问控制，不能只靠前端限制文件名。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q19 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：复用文件包含基线并观察限制

操作

先按文件包含题的方式测试页面参数，再尝试常见源码读取方式，记录哪些被拦截。

预期观察

普通页面可被包含，但部分 wrapper 或敏感路径被过滤。

简要分析

题目通常在已有 LFI 基础上增加过滤，引导使用日志污染等二阶利用。

为什么要这样做

先确认旧方法失败原因，避免学生误以为题目不可解。

当前步骤验证什么假设

验证当前包含点仍存在，但直接源码读取受限。

Step 2：寻找可控且会落盘的日志入口

操作

观察请求头、访问日志提示或报错信息，判断 User-Agent 等字段是否会写入服务器日志。

预期观察

服务端访问日志会记录部分客户端可控内容。

简要分析

日志污染依赖两件事：攻击者可写入日志，文件包含点可包含日志。

为什么要这样做

这一步把输入点和包含点连接起来。

当前步骤验证什么假设

验证是否存在可被攻击者间接写入的服务器端文件。

Step 3：写入无害执行探针

操作

通过可控请求头写入一个最小化、可识别的服务端代码探针，然后包含日志文件观察效果。

预期观察

如果包含日志时探针被解释执行，会出现预期标记或输出。

简要分析

先用无害探针确认执行链，避免一开始读取敏感目标时无法区分失败原因。

为什么要这样做

日志文件本来是数据；被 include 后，其中的代码片段会进入解释器。

当前步骤验证什么假设

验证日志污染内容是否能通过 LFI 转化为代码执行。

Step 4：用已验证执行链读取题目结果

操作

把探针替换成读取题目目标内容的逻辑，再通过包含日志触发执行。

预期观察

响应中出现题目结果或可用于定位结果的输出。

简要分析

最终动作只是复用已经验证的日志执行链。

为什么要这样做

分阶段构造能让学生理解从文件包含到代码执行的升级过程。

当前步骤验证什么假设

验证日志污染链是否能完成题目要求。

Step 5：总结限制和修复

操作

记录修复方向：包含路径白名单、日志文件禁止被 Web include、关闭危险解释入口、正确转义日志内容。

预期观察

学生能区分 LFI、日志污染和 RCE 之间的关系。

简要分析

本题不是单一参数绕过，而是两个弱点组合。

为什么要这样做

修复需要同时处理文件包含边界和日志可执行问题。

当前步骤验证什么假设

验证教学材料是否明确二阶利用链路。

执行证据摘要

Q19 direct backend 执行证据

  • 用 User-Agent 写入 PHP payload 到 access.log：HTTP 200
  • include access.log 触发执行：HTTP 200
  • flag 已写入 work/solve/19/flag.txt

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
