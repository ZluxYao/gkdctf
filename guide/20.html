Q20 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面涉及文件包含或 PHP wrapper 时，先确认参数是否进入 include/readfile，再区分源码读取、日志污染和 data wrapper 执行。
  • 题面涉及命令执行时，先确认参数是否拼接到系统命令，再判断是否有回显、过滤字符和可用外带通道。
  • 题面涉及文件读取或路径参数时，先确认根目录限制，再逐层尝试相对路径，判断是否能越过预期目录。

建议观察清单

  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用普通页面名做基线，再测试 wrapper、日志路径或可控请求头，观察响应是否包含源码、执行输出或错误信息。
  • 用基线参数确认正常功能，再用安全的分隔符测试是否改变输出；无回显场景优先找日志或 OOB 记录通道。
  • 比较正常文件、缺失文件和上级目录文件的响应差异，用错误信息缩小真实路径。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • 文件包含题要区分 LFI、RFI、源码泄露和代码执行，修复重点是白名单和禁用危险 wrapper。
  • 命令注入的核心是服务端把用户输入当作 shell 语法解释，修复应避免 shell 拼接并使用参数化 API。
  • 路径穿越题的关键是规范化路径和访问控制，不能只靠前端限制文件名。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q20 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：观察远程资源加载参数

操作

访问题目功能，找出用于加载插件、模板或远程内容的 URL 参数，并用正常值建立基线。

预期观察

正常资源可被加载；明显外部 HTTP/HTTPS 地址可能被提示拦截。

简要分析

远程文件包含常见防护会黑名单化某些协议，但不一定覆盖所有 wrapper。

为什么要这样做

先确认参数语义是“加载资源”，而不是普通跳转。

当前步骤验证什么假设

验证用户是否能控制服务端加载的资源地址。

Step 2：测试协议过滤边界

操作

分别尝试不同协议形式和大小写、编码变化，记录哪些被拦截、哪些继续进入加载逻辑。

预期观察

响应差异显示过滤主要关注特定协议前缀。

简要分析

黑名单过滤容易遗漏等价协议或非网络型数据包装器。

为什么要这样做

用对照测试可以推断过滤规则，而不是直接背 payload。

当前步骤验证什么假设

验证服务端是否采用协议黑名单而非严格白名单。

Step 3：确认服务器解释包含内容

操作

构造最小化的内联内容，让服务端把它当作待包含资源处理，观察是否产生预期输出。

预期观察

如果输出来自内联内容，说明包含发生在服务器端解释环境。

简要分析

RFI 的危险在于攻击者控制的内容进入解释器。

为什么要这样做

先验证解释能力，再读取题目结果，能定位失败环节。

当前步骤验证什么假设

验证非 HTTP/HTTPS 形式是否仍可触发包含执行。

Step 4：用受控资源完成题目目标

操作

在已验证的可加载形式中放入读取目标内容的逻辑，提交后观察响应。

预期观察

响应返回题目目标结果或其编码形式。

简要分析

这一阶段是利用已确认的 RFI/wrapper 行为完成结果读取。

为什么要这样做

学生需要理解绕过点是协议过滤，不是凭空出现的后门。

当前步骤验证什么假设

验证服务端是否执行了受控资源中的读取逻辑。

Step 5：归纳修复策略

操作

记录：禁用远程包含，使用固定插件白名单，不把用户 URL 交给 include，过滤前做规范化。

预期观察

学生能说清楚为什么仅禁止两个协议不够。

简要分析

根因是把用户输入当作代码资源加载，并依赖脆弱黑名单。

为什么要这样做

修复必须从设计上移除用户可控 include。

当前步骤验证什么假设

验证最终指引是否解释了协议绕过和 include 风险。

执行证据摘要

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
