## 漏洞本质

两个独立误把关的漏洞，合起来就是核弹：

| 漏洞 | 位置 | 独立影响 |
|---|---|---|
| 存储型 XSS | `/board` 评论未转义 -> `<script>` 执行 | 窃取全员受害者 cookie |
| CSRF Token 可同源读 | `/admin/console` 的 token 嵌在 HTML，同源 JS 能 `fetch()` | 存在方法穿透 CSRF |
| 标准纯 CSRF（没 XSS 时） | - | Token 进不去，失效 |

射击链：XSS 进右 admin 浏览器 -> 代码 JS 同源 fetch -> 窃 token + 窃 cookie -> 用同一把钥匙去打受重保护的 `/admin/grant`。

## 关键前置知识

XSS 三开对比：

| 类型 | 存储位置 | 触发方式 | 本题 |
|---|---|---|---|
| 反射型 | URL 参数 | 受害者点恶意链接 | × |
| 存储型 | 数据库 / 文件 | 浏览页面自动触发 | ✓ |

| 类型 | 存储位置 | 触发方式 | 本题 |
| --- | --- | --- | --- |
| DOM 型 | 前端 JS 注入 | 前端操作 | × |

CSRF Token 两种预存模式:

| 模式 | 存储 | 疑问点 |
| --- | --- | --- |
| 同步器模式（Synchronizer Token，本题） | 服务端 session 里挂一份，模板当场嵌到 `<form>` | 模板能读，同源 JS 也能读 —— 这条链被 XSS 方向打暴 |
| 双交 cookie | cookie + 表单各带一份 token | cookie 有 HttpOnly 也没用—模板一写到 HTML 中就瞎 |

Same-Origin Policy 下 fetch 的踩黑行为:

- 同源 fetch 默认 `credentials: 'same-origin'`，admin cookie 自动随行
- 能读响应体——这正是 token 被掘走的关键
- 哪怕有 `HttpOnly`、`SameSite=Strict`、`Secure` 至尊，在同源 fetch 下仍移动完美传没

> Important
>
> 为什么 CSRF Token 在 XSS 面前彻底失效:
>
> - Token 的“秘密”假设是“攻击者在另一个站点，不知道 token”。
> - XSS 在攻击目标内部跑，与正常页面共享 DOM、cookie、fetch 权限——假设从根上坍。
> - 故真正的修复是除掉 XSS（CSP、转义、对比白名单）而不是加强 Token。

| 问题 | 根因 | 解决 |
|---|---|---|
| grep LEAK1 出来的是 payload 源码 | /board 自己也输出了 payload（符合标准） | `tail -1` 取最后一条；用更强的标签区分 |

# XSS + CSRF 组合谱

Important

1. **fetch-drone（本题）**：XSS 里现抠 token 代 admin 发请求
2. **键盘记录 / 密码窥探**：给所有 input 绑 onkeyup，收点击输入
3. **Service Worker 剧持**：注册恶意 SW → 拦所有 fetch → 长驻住
4. **Keylogger + CSRF Chain**：先偷委任令虹 再弥长目异授权
5. **Content-Spoof（奇诗人替代 admin 面板）**：给 admin 看伪造面板，诱黏他亲自点链接
6. **CSP 绕过**：若 CSP nonce 嗑开，让 `<script>` 再现的方法：`<base href>` 劫持 / `<link rel=import>`
