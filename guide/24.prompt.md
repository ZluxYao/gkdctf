## 漏洞本质

`"字符串拼接"` vs `"安全路径解析"`：

`/var/www/uploads/` + `../../../../flag.txt` = (字符串拼) = `/var/www/uploads/../../../../flag.txt`

对操作系统（以及 PHP 的 `readfile` 底层 `open(2)`）而言：

- 内核解析路径时，`..` 是一个目录项，代表父目录
- `uploads/..` = `/var/www/`，`/var/www/..` = `/var/`，`/var/..` = `/`
- 拼完的 `/var/www/uploads/../../../../flag.txt` 和 `/flag.txt` 是同一个 inode

这条漏洞的 3 个关键错位：


## Page 97

| 错位 | 开发者以为 | 现实 |
|---|---|---|
| ① 输入的边界 | "用户只会输入文件名" | 用户可以输入任何字符串，包括 `..` `/` |
| ② base 的作用 | "前缀 base 就够沙箱了" | 字符串前缀 ≠ 目录沙箱，`..` 能穿透 |
| ③ file_exists 的语义 | "存在就是合法" | 它只检查文件是否存在，不检查是否在 `base` 之内 |

\

**正确的判定应当是：** 先把拼好的路径 `realpath()`，再判断是否以 `realpath($base)` 为前缀。

**Important**

`basename()` 的真伪：

代码里用了 `basename($name)` 作为下载文件名——但它只影响 HTTP 响应头里的 `filename="..."`，不影响 `readfile` 读哪个文件。这是新手最容易误读的"看似防御"：`basename` 防的是输出而不是输入。

`../` 序列的 3 种语义形态：

| 形态 | 示例 | 何时折叠 |
|---|---|---|
| 纯字符串 | `uploads/../../flag` | 永远不折叠（只有你 `realpath` 它） |

`../` 层数的经验公式：

> 层数 ≥ `base` 的目录深度 即可越到根（多了无所谓，`/../` = `/`）。

## 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| curl 直接 `?file=../../../../flag.txt` 没出内容 | 某些 shell / curl 会 URL-encode 但浏览器不会 | 加 `-g` 或 `--path-as-is`；或干脆浏览器地址栏访问 |
| 加了 `../` 但返回 404 | `file_exists` 为真需要目标存在；拼错路径必 404 | 先用 `/etc/passwd` 等“必存在”的系统文件校准 |
| 返回空文件 | 目标是目录（`is_dir` 拦） | 确保目标是文件；用 `/proc/self/maps` 做非空验证 |
| 某些代理自动折叠 `../` | HTTP 客户端/反代将 URL 里的相对段归并后再发 | `curl --path-as-is` 保留原始字节 |
| `basename()` 干扰 | 以为它过滤了输入 | 它只影响 响应头的建议文件名，不影响 readfile 读哪个文件 |
| 本地 base64 不支持二进制 flag | flag 有奇怪字符 | 本题是纯 ASCII，无此问题；对二进制文件走 `?file=../a.bin` 时需 `-O` 落盘 |
| 浏览器直接显示而不下载 | `Content-Disposition: attachment` 没被浏览器尊重 | 无所谓，flag 内容能看到就行 |

## 路径穿越进阶 8 谱

Important

1. **绝对路径替代**：若 base 用 `$base . $file` 拼，输入 `/etc/passwd` 可能越界；若用 `join(base, user)` 要看语言实现
2. **URL 二次编码**：`%252e%252e%252f` = `%2e%2e%2f`（经代理解码一次）= `../`（经后端再解码一次）→ 见 Q25
3. **反斜杠 Windows**：`..\..\flag.txt` 在 Windows 或用 `\` 作为分隔符的库里生效
4. **UTF-8 覆写**：`%c0%ae%c0%ae/` 历史版 Apache/IIS 过滤失效
5. **Null byte 截断**：PHP < 5.3 / C 风格 API 遇 `%00` 截断，`?file=flag.txt%00.png` 绕扩展名白名单
6. **软链接穿出**：靶机 uploads 下的软链接指向 `/` → 即便有 realpath 校验也能穿出
7. **路径规范化差异**：`./` `/.//` `a/../../b` 在不同库下规范化结果不同，WAF 与后端规范不一致即绕
8. **Zip-Slip / Tar-Slip**：压缩包内含 `../a`，解压到 base 外 → 见 Q26

## 关键修复

Important

**修一（核心银弹）：realpath + 前缀校验**

[代码块边界]php
$base = realpath('/var/www/uploads/');              // 规范化后的绝对路径
$real = realpath($base . DIRECTORY_SEPARATOR . $name); // 先拼后归一

// 两重校验：存在 + 必须落在 base 内
if ($real === false || strpos($real, $base . DIRECTORY_SEPARATOR) !== 0) {
    http_response_code(404); exit('bad file');
}
readfile($real);
[代码块边界]

- `realpath` 把所有 `..` 折掉，落在文件系统真实位置
- 再用目录前缀 + `DIRECTORY_SEPARATOR` 防止 `/var/www/uploads_x/` 这类“前缀包含但不真在目录下”的 bypass

**修二：白名单文件名**

[代码块边界]php
$allowed = ['report_a.txt','report_b.txt','summary.txt'];
if (!in_array($name, $allowed, true)) exit('bad file');

[代码块边界]

- 只认固定文件名，`..` 直接被 `in_array` 拒
- 适合文件列表稳定、数量有限的场景
