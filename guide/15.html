Q15 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面强调验证码时，先判断验证码是否绑定服务端 session、是否一次性失效，以及是否存在前端校验或弱字符集。
  • 题面出现 token/JWT 时，先区分 token 的用途：身份声明、一次性登录令牌、CSRF token 或接口访问凭证。
  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面出现 bot、XSS、CSRF 时，先判断谁会访问页面、访问时携带什么身份、脚本能否同源读取敏感页面。
  • 题面涉及文件读取或路径参数时，先确认根目录限制，再逐层尝试相对路径，判断是否能越过预期目录。

建议观察清单

  • 保持同一个会话访问登录页、获取验证码图片，再提交一次错误密码；如果返回密码错误而不是验证码错误，说明验证码已被服务端接受。
  • 观察 token 的获取接口、提交位置、过期行为和错误响应，再判断能否伪造、复用、侧信道推断或同源读取。
  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用最小化回显或回连验证脚本是否执行，再设计读取 token、代发请求或状态确认链路。
  • 比较正常文件、缺失文件和上级目录文件的响应差异，用错误信息缩小真实路径。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 验证码的安全性不只取决于图片难度，还取决于服务端是否正确绑定会话、请求和失败次数。
  • 令牌类题目要关注签名算法、密钥强度、生命周期、绑定关系和响应时间等旁路信息。
  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • CSRF token 防的是第三方站点伪造请求；如果存在同源 XSS，脚本可能读取页面里的 token 并以受害者身份发起请求。
  • 路径穿越题的关键是规范化路径和访问控制，不能只靠前端限制文件名。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q15 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：黑盒观察普通用户权限

操作

以普通学生身份登录，访问个人信息、后台入口和题目提供的提交入口。

预期观察

普通用户不能直接访问管理员功能，但可以把一个 URL 提交给后台访问。

简要分析

这类题通常考查管理员浏览器携带自身 Cookie 访问攻击者提供的链接。

为什么要这样做

先确认是否存在 bot/管理员访问通道，才能判断是否是 CSRF 类题目。

当前步骤验证什么假设

验证普通用户是否可以诱导管理员会话发起请求。

Step 2：寻找会改变状态的 GET 请求

操作

在自己的账号上测试个人资料或改密功能，观察是否存在仅靠 GET 参数就能改变状态的接口。

预期观察

如果访问某个链接后账号状态发生变化，说明该接口缺少方法约束和 CSRF 校验。

简要分析

CSRF 的关键是让受害者浏览器自动带 Cookie 发起状态改变请求。

为什么要这样做

先在自己账号上验证副作用，避免直接对管理员做不可解释的尝试。

当前步骤验证什么假设

验证目标接口是否能被跨页面链接触发并产生状态变化。

Step 3：构造管理员会访问的状态改变链接

操作

把已验证的 GET 状态改变请求改成作用于当前登录用户的形式，然后提交给后台访问队列。

预期观察

提交后需要等待管理员 bot 访问；成功时管理员账号状态会被改变。

简要分析

如果接口根据 Cookie 中的当前用户决定对象，管理员访问同一链接就会作用到管理员自己。

为什么要这样做

这一步把“普通用户可改自己”扩展为“诱导管理员改管理员”。

当前步骤验证什么假设

验证 GET 型 CSRF 是否能借助管理员 bot 影响高权限账号。

Step 4：用受控状态验证管理员权限

操作

在等待 bot 处理后，用受控的新状态尝试进入管理员功能；失败则检查链接参数、bot 队列和等待时间。

预期观察

成功时可以访问原本禁止的管理页面。

简要分析

最终访问管理页只是验证 CSRF 是否成功，不应作为第一步直接猜管理员凭据。

为什么要这样做

把等待和验证分开，能定位失败是 bot 未访问还是状态改变请求无效。

当前步骤验证什么假设

验证管理员状态是否确实被诱导请求改变。

Step 5：总结防护要点

操作

在解题记录中说明：状态改变操作应使用 POST、加入 CSRF token、校验 Origin/Referer，并要求旧口令或二次确认。

预期观察

学生能看到漏洞来自请求语义和缺失 CSRF 防护，而不是管理员口令泄露。

简要分析

GET 请求不应承担敏感状态改变，是 Web 安全基础约束。

为什么要这样做

修复建议帮助学生把题目经验迁移到真实开发。

当前步骤验证什么假设

验证指引是否先讲观察和诱导链路，再讲最终结果。

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
