# 关键前置知识

CSRF (Cross-Site Request Forgery) 三道防御:

| 防御 | 原理 | 绕过场景 |
|---|---|---|
| CSRF Token（同步器模式） | 服务端下发一个随机 token，必须跟请求一起交 | 第三方 不知道 token 无法构造 |
| Double-Submit Cookie | 在 cookie 和请求头同时带 token，服务端比对 | 第三方读不到 cookie |
| SameSite Cookie | 浏览器级别— `Lax` / `Strict` 来自外站不带 cookie | Chrome 80+ 默认 Lax，但 GET 的顶级跳转仍放行 |
| Referer/Origin 检查 | 请求来源非本站就拒绝 | 需配合前三项，Referer 单独拎出并非铁片 |
| 二次认证（旧密 / OTP / 短信） | 敏感跳功能要求双因子 | 与其他防御叠加 |

SameSite Cookie 的演进:

| 版本 | 行为 | 对 CSRF 的影响 |
|---|---|---|
| 无 SameSite | 跨站随意带 cookie | 全见 CSRF |
| `SameSite=Lax`（现代默认） | 跨站的 GET 顶级导航带，POST / iframe / AJAX 不带 | GET CSRF 仍然能走超链接/当 iframe src |
| `SameSite=Strict` | 跨站 完全不带 | CSRF 几乎全挂 |
| `SameSite=None; Secure` | 显式允许跨站 | 需自己上其他防御 |

Important

本题的白带特殊性：攻击者不是在受害者浏览器里触发，而是 服务端自带的 bot 直接用 requests 库访问——这种场景是标准 CSRF 的变体，SameSite 全失效（bot 本身就是同源），Referer 校验也拦不住（bot 可以伪造）。这在现实系统常见于：

- 后台审核系统（客服点链接触发操作）
- 消息背板驱动替你点同源链接（URL Preview Bot）
- SSRF + 内网管理后台

# 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| 提交 URL 后 admin 密码没变 | bot 周期 15s，没等足 | sleep 至少 16s，保险至 18–20s |
| payload URL 使用站外域名 | bot 只收 `/` 或 `http://127.0.0.1/` | 换成相对路径 |
| admin 密码改成了但自己登不上 | 混在旧会话 | `curl -c` 换新 cookie jar 重新登录 |
| GET 会不会之未带 cookie | 现代 session 默认 SameSite=Lax，本题 bot 是服务端同源请求 | 与 SameSite 无关，因为 bot 不是浏览器跨站 |
| 只改了密码但登不进 /admin | admin 账号被锁定 / 白名单 IP | 本题无此限制，如遇则改成读 email / reset token |
| payload 用中文 / 空格 | URL 未编码 | 用 `%20` / `%E4%B8%AD` |

# CSRF 通用套路

Important

1. GET 型：`<img src="http://bank/transfer?to=X&amt=1000">` ——看图就躺钱
2. POST 型：恶意页放隐藏 form + auto-submit JS，形如 `<form action="..." method=POST><input name=to value=X>...</form><script>document.forms[0].submit()</script>`
3. JSON 型：Content-Type: `application/json` 跨站会触发 preflight——用 `text/plain` + 在 body 里塑 JSON 绕过 CORS
4. 同站 CSRF（本题）：中继机器人 / URL Preview / 后端审核 / Webhook 消费者 —— SameSite 无效，需走“黑单内网 IP” + “要 CSRF Token”
5. CSRF + XSS 链：XSS 征召受害者浏览器 → fetch 自己所在站 /csrf-token → 再发伪请求—能穿透 Token 和 SameSite
6. 登录 CSRF：恶意页面给受害者用我的账号登录... 后续故意记录受害者行为或拿到便捷接口 token
