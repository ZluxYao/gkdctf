# 题目背景

> Note

MiniShop 迷你商城：

- 首页 /：展示当前余额（初始 ¥100）+ 3 种商品
  - 可乐 `coke` ¥5
  - iPhone `phone` ¥9999
  - 机密物料 `flag` ¥100000（买下返 flag）
- 购买接口：`POST /buy`，字段 `item` + `qty`
- 余额重置：`GET /reset`（重置到 ¥100）
- 余额永远 ≥ 0 才能买（后端拒 `balance not enough`）
- Flag 藏在"购买 flag 商品成功"的响应 JSON 里

正常流程：余额 ¥100 根本买不起 ¥100000 的 flag 商品。但服务端的扣款逻辑有个**整数域下界未校验**的经典逻辑漏洞。

# 漏洞本质

业务逻辑漏洞（Business Logic Vulnerability）是 OWASP Top 10 2021 的 A04: Insecure Design 核心品类之一。它不是 SQL 注入/XSS/命令注入这些"技术漏洞"，而是开发者在业务建模时就漏了一个不变量假设。

本题的漏洞核心：

[代码块边界]php
// 有漏洞的代码模型
$qty   = (int)$_POST['qty'];       // ← 只做了整型转换，没校验范围
$price = $ITEMS[$item]['price'];
$total = $qty * $price;            // qty=-30000 * 5 = -150000

if ($_SESSION['balance'] < $total) { // 🚨 只检查下界，假设 total > 0
    exit(json(['msg' => 'balance not enough']));
}
$_SESSION['balance'] -= $total;      // balance -= (-150000) = balance + 150000

if ($item === 'flag') {
    echo json(['flag' => $FLAG]);    // 😈 钱够就送 flag
}
[代码块边界]

\

**开发者心智里的"隐式前置条件"：**

> "qty 是用户买多少件一用户怎么可能买负数件？"

这个假设在 UI 层是对的（表单默认 value=1, min=1），但 HTTP 接口**完全在客户端可控**。攻击者只要直接按 `curl` 绕过前端，`qty=-20000` 就会穿过服务端。

\

## 为什么业务逻辑漏洞很难防？

| 维度 | 技术漏洞（SQLi/XSS） | 业务逻辑漏洞 |
| --- | --- | --- |
| 识别模式 | 有固定的字符串模式（`' OR 1=1`） | 没有任何特征，每个业务都不同 |
| WAF 能否拦 | 能 | 完全不能 |
| 自动化扫 | SAST/DAST 工具链成熟 | 工具无法推断业务语义 |
| 防御手段 | 参数化查询、编码、CSP | 只能靠领域建模 + 代码 review + 测试用例 |
| 修复定位 | 改一行调用 | 改业务流程 |
| 真实世界 | 占公开漏洞 20%-30% | 占高危漏洞 60%+（BugCrowd 2024 数据） |

> Important

**本题一句话心智模型：**

"用户输入的每个数值都必须同时校验类型、范围、精度、上下界"一仅 `(int)` / `(float)` 转型远远不够，必须显式 `if ($qty < 1 || $qty > MAX)` 双边校验。

## 常见坑

| 问题 | 根因 | 解决 |
| --- | --- | --- |
| qty=-1 返 `balance not enough` | 后端用 `abs($qty) * $price` 校验 | 换大负数触发整数溢出 |
| 响应 `ok:true` 但余额不变 | 幂等/缓存拦了重复请求 | 加随机参数破缓存 |
| balance 负数但仍 `balance_not_enough` | 后端用 `abs(balance) < total` 错判 | 试 `qty=0` 观察日志 |
| `qty=-30000` 直接 4xx | 后端做了下界校验 | 换上界溢出 / 浮点精度 |
| session 不跨请求 | 没带 cookie | `-b /tmp/q32.ck` |
| 多次 reset 之后 flag 没了 | 本题 flag 每次 session 实时读 | 不影响 |
| 服务端记账用浮点 | `0.1 + 0.2 ≠ 0.3` | 所有金额应存 “分”（整数） |

## 业务逻辑漏洞进阶 10 谱

Important

1. 负数金额/数量【本题】：单边校验
2. 整数溢出：`int32 + int32` 变负
3. 浮点精度：四舍五入吃差价
4. 竞态条件（Q34）：并发请求绕库/余额检查
5. 价格参数伪造：前端传价格，后端不重查
6. 币种/汇率伪造：支付时切便宜币种
7. 优惠券叠加/重用：未做 owner + 订单去重
8. 订单状态竞态：先支付再发货校验、多状态跳转
9. 重量/退款超额：退款金额 > 实付
10. 二阶逻辑：A 流程看起来安全但 B 流程叠加出现破绽

### 修三：二次扣款校验（防负数）

[代码块边界]php
if ($qty <= 0) { exit('qty must be positive'); }
if ($total <= 0) { exit('total must be positive'); }
if ($_SESSION['balance'] < $total) { exit('insufficient balance'); }

// 事务扣款
DB::transaction(function () use ($user, $total) {
    $balance = DB::lockForUpdate()->select('balance')->where('id', $user->id);
    if ($balance < $total) throw new Exception('insufficient');
    DB::update(['balance' => $balance - $total])->where('id', $user->id);
});
[代码块边界]

### 修四：DTO + Schema 校验

- Laravel：`FormRequest` 强类型 + rules
- Spring：`@Valid @Min(1) @Max(1000)` 注解
- Express：`Joi.number().integer().min(1).max(1000)`
- FastAPI：`Field(..., ge=1, le=1000)`

### 修五：模糊测试 / 属性测试

- 用 Hypothesis / QuickCheck 随机生成 qty 和 price
- 断言"用户余额变化方向与 qty 符号相反"
- 断言"任何交易后 balance >= 0"

### 修六：监控告警

- 日志记录所有 `qty < 0` 或 `total < 0` 的请求
- 实时告警（正常业务绝不该出现负数流水）

### 修七：幂等 + 订单号 + 状态机

- 每笔交易先生成订单号，支付后改状态
- 同一订单号不能重复扣款
- 改 qty / price 之前必须检查订单状态

### 修八：合规

- PCI-DSS（支付卡行业）、ISO 27001 都要求 金额必须用 `fixed-point` 算术
- 审计日志不可删，所有金额变化可追溯
