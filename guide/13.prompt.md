# 漏洞本质

JWT HS256 的致命前提： `HS256 = HMAC-SHA256` 是对称签名，“签”和“验”用的是同一个密钥。一旦密钥为弱口令/默认值，攻击者仅需一条合法 JWT 就能离线爆破（不过靶机、不留日志），拿到密钥 = 任意身份的签发权。

| 环节 | 配置 | 缺陷 |
|---|---|---|
| 密钥 | `qwerty` | Top-100 字典秒杀 |
| 签名算法 | `HS256` | 对称，攻防同权 |
| 负载 | 明文 base64，能直接读 `role` | 知道该改哪个字段 |

`Caution`

Flag 位置：`/admin/flag` 验签 + `role=admin` 删两层 → 返回 `{"flag":"[FLAG 占位]","ok":true}`。

# 关键前置知识

JWT 结构（三段 base64url，以 `.` 分隔）：

[代码块边界]text
<header>.<payload>.<signature>
header    = base64url(json)    |alg/typ|
payload   = base64url(json)    |业务字段（sub, role, exp…）|
signature = base64url( HMAC-SHA256( header + "." + payload , secret ) )
[代码块边界]

HS256 的碰撞矩阵：

| 缺陷 | 绕过 |
|---|---|
| `secret` 为弱口令/默认值 | 离线 HMAC 字典爆破 |
| 使用 `alg:none` | 缺固库需 `require` alg，未声明时可用 `{"alg":"none"}` + 空签名 |
| RS256 / HS256 混淆 | 原站用 RS256，改成 HS256 + 用公钥当密钥（公钥可公开拿） |
| `kid` 字段 SQLi / 路径穿越 | 让服务端拉取攻击者伪造的密钥 |
| `jwk` / `jku` | 让服务端拉取攻击者控制的公钥 JSON |

# 解题思路推导（从看到题面到拿 Flag）

这一段写给第一次碰 JWT 题的选手——为什么是这条路，不是别条路。

## 阶段 ①：读题面，抽约束

题面给的信息，整理成“帮”和“拦”两栏：

| 条件 | 帮 / 拦 | 推导出的信号 |
|---|---|---|
| 有 `guest/guest` 测试账号 | 帮 | 我能无门槛拿一个合法 JWT（不用爆登录） |
| 管理员密码未公开 | 拦 | 走“爆 admin 密码”这条路被排死 |
| token 负载包含 `role` 字段 | 帮 | 最大剧透——身份是写在 payload 里，而不是每次查库 |
| 算法 明写 HS256 | 帮 | 对称签名，拿到 secret 就能随便签 |
| 受保护接口 写清要 `role=admin` | 帮 | 知道改哪里 |

四行“帮”拼起来是一个完整的利用链模型：

> ① 用 guest 登录 → ② 拿到合法 JWT → ③ JWT 里 role=user，改为 admin → ④ 但改完签名不对，要用 secret 重签 → ⑤ secret 未知，找绕过

于是问题被压缩为一句话：**我怎么在不知道 secret 的情况下重签一个 JWT？**

## 阶段 ②：三条候选路径排序

拿不到 secret 的情况下，常见有三条路：


## Page 47

| 路径 | 原理 | 本题可行性 |
|---|---|---|
| A. `alg:none` 绕过 | 个别库看 `alg:none` 就不验签 | 盲猜，先尝试一下即可确认 |
| B. RS256 ↔ HS256 混淆 | 原站用 RS256，我改 HS256 + 用公钥当密钥 | 本题本身就是 HS256，没有 RS256 混淆空间 |
| C. HS256 离线字典爆破 secret | secret 弱 → 用合法 JWT 本地比对 HMAC | 拿到了 guest JWT，离线无限尝试 |

Tip

为什么优先试 A 又为什么一般不中：

- 成本低：改一行 header 就行，不用爆密钥
- 但主流 `jsonwebtoken` / `pyjwt` 自 2019 年起默认拒绝 `alg:none`，除非开发者手动去 `require`，都是一提就挂
- 验证成本也低（30 秒），先打 A 再备 C 是正确姿势

先快速试 A。把 header 改成 `{"alg":"none","typ":"JWT"}`，payload 改成 `{"sub":"admin","role":"admin"}`，signature 段留空，三段拼 `header.payload.`（末尾有个点但不留值）。

本题测试结果：服务端 返回 `401`，表明 `alg:none` 被拿下 → 走 C 路线。

## 阶段 ③：HS256 爆破的数学直觉

为什么 HMAC 能离线爆破？

[代码块边界]text
1            header.payload （明文，能从 JWT 里抽出来）
2                  ↓ HMAC-SHA256
3    signature = H(key, header.payload)    （已知，第三段 base64url 解出）
[代码块边界]

将已知量代进去：

- 已知：`msg = header + "." + payload`
- 已知：`signature = <JWT 第三段 base64url 解>`
- 唯一未知：`key`
- 验证一个 `key` 候选只需 一次本地 HMAC-SHA256 运算 + 一次二进制比对
- CPU 上 Python 一秒几十万次，`hashcat -m 16500` GPU 上可千万/秒

结论：rockyou.txt（1400 万行）全30 秒内结束。而且全程不过靶机，IDS/WAF 看不到。

## 阶段 ④：拿到 secret 后的重签模板

secret 一拿到，整个 JWT 签发权就归我了。改什么 payload 都行。

[代码块边界]text
1  伪造模板:
2  header  = {"alg":"HS256","typ":"JWT"}
3  payload = {"sub":"admin","role":"admin"}   ← 把 guest/user 改 admin/admin
4  sig     = HMAC-SHA256( base64url(header) + "." + base64url(payload) , secret )
5  token   = base64url(header) + "." + base64url(payload) + "." + base64url(sig)
[代码块边界]

这里有两个经典坑，踩过的人永远记得：

Warning

坑 1：`json.dumps` 默认在 `,` / `:` 后加空格，产生 `{"sub": "admin", …}`，而服务端给我们的 token 是 `{"sub":"admin",…}`。两个 `msg` 不同 → 算出来的 signature 天地之差 → 服务端直接拒收。

修复：`json.dumps(obj, separators=(',', ':'))`

坑 2：base64 标准编码会生 `+`/`/`/`=`，但 JWT 规范用 `base64url` 且删掉 padding `=`。

修复：`base64.urlsafe_b64encode(x).rstrip(b'=')`

h, p, s = JWT.split(".")
msg = f"{h}.{p}".encode()
target = base64.urlsafe_b64decode(s + "="*(-len(s)%4))

for w in open("rockyou.txt"):
    w = w.strip()
    if hmac.new(w.encode(), msg, hashlib.sha256).digest() == target:
        print("SECRET =", repr(w)); break

## Forged Token 拆解

| 片段 | 值（base64url 前） | 说明 |
|---|---|---|
| header | `{"alg":"HS256","typ":"JWT"}` | 与原 token 保持一致，不能随便改 alg |
| payload | `{"sub":"admin","role":"admin"}` | 改的重点：`role:user` → `role:admin` |
| signature | `HMAC-SHA256(h.p, "qwerty")` | 用爆出来的密钥重签 |

Important

两个容易翻车的细节:

1. `json.dumps` 必须带 `separators=(',',':')`——消除空格，否则签出来的 signature 与实际 payload 不匹配，服务端一比就挂。
2. base64url 去掉尾巴 `=`（`rstrip(b'=')`）——JWT 规范不尾随 padding。

# 常见坑

| 问题 | 根因 | 解决 |
|---|---|---|
| 伪造 token 服务端 401 | JSON 序列化带了空格 | `separators=(',',':')` |
| base64 包含 `/` `+` | 没用 base64 url 变体 | `base64.urlsafe_b64encode` |
| 签名尾巴有 `=` | 未剔 padding | `rstrip(b'=')` |
| 注入 `{"alg":"none"}` 没用 | 库默认要求 `alg:HS256` | 换 HS256 弱密钥路线 |
| `hashcat -m 16500` 一直 0 Hits | JWT 格式不对 / 字典过小 | 取 JWT 全文直接写入 `jwt.txt`，换 rockyou.txt |

# JWT 攻击通用套路

> Important

1. 任何 JWT 拿到手先解开看 payload - `role/admin/user_id/scope/iss` 字段在哪
2. HS256 看到先试弱口令字典（`secret` / `123456` / `jwt` / 项目名 / `changeme`），hashcat 30 秒见真章
3. RS256 看看有无 `/.well-known/jwks.json`，不在也可看极少数站通过 `x5c` / `jwk` header 自带公钥
4. 检查是否有 `alg` 混淆：把 `alg` 从 `RS256` 改 `HS256`，用公钥当密钥重签
5. 看 `exp` `iat` `nbf` 是否实现正确，有的站只签不验 `exp` -> 过期令牌永远可用
6. 随手尝试 `{"typ":"JWT","alg":"none"}` + 空签名：官方库都已默认拒绝，但少数自写库仍会中招

# 关键修复

> Important

修一：密钥用 256 位随机，绝对不进代码仓库：

[代码块边界]bash
openssl rand -base64 48
# 放到 JWT_SECRET 环境变量 / Vault
[代码块边界]

修二：验签时强制指定 algorithm（缓灾 `alg:none` / RS256->HS256 混淆）：

[代码块边界]python
jwt.decode(token, SECRET, algorithms=['HS256'])  # 显式传 algorithms
[代码块边界]

修三：敏感业务切换 RS256（非对称，私钥在服务端、客户端只拿公钥，泄露公钥不影响安全）：

[代码块边界]python
jwt.encode(payload, PRIVATE_KEY, algorithm='RS256')
jwt.decode(token, PUBLIC_KEY, algorithms=['RS256'])
[代码块边界]

修四：blacklist/rotation：结合服务端 `jti` 黑名单支持“独木登出”；关键账户轮换 secret。

修五：不把整个用户画像塞 payload：payload 是 base64 明文，任何持有方都能读，不要装账号余额/手机号这种数据。
