## 漏洞本质

POST 型 CSRF = 两个“独立看没事、合到一起炸锅”的特性：

1. `POST /transfer` 仅靠 `session` 鉴权，无 CSRF Token 无 Referer 无 Origin 检查
2. `/pages/host` 让任意低权用户把 `HTML + JS` 托管到同源路径下

于是：攻击者写一段 auto-submit form 的 HTML → 托到同源 → bot 来访问 → JS 执行 `form.submit()` → bot 带 admin cookie 打 POST `/transfer` → 转账得逞 → inbox 送 flag。

| 缺失的防御 | 影响 |
|---|---|
| × 无 CSRF Token | POST 表单任意构造 |
| × 无 Origin / Referer | 不管请求来源 |
| × 无交易密码 / 密钥产出 | 转账仅凭 session 便执行 |
| × `/pages/host` 无沙箱 / CSP | 恶意 HTML 直跳 |
| ✓ session cookie 默认 `SameSite=Lax` | 对本题完全无效，bot 本身在同源，不踏 SameSite 诡计 |

# 关键前置知识

GET vs POST 型 CSRF 对比:

| 维度 | GET CSRF（Q15） | POST CSRF（本题） |
|---|---|---|
| 触发方式 | 一条 URL / `<img src>` | auto-submit `<form>` + JS |
| 需要执行 JS | 不需 | 需（才能自动提交） |
| bot 类型 | requests / curl 即可 | 真浏览器（Chromium/Puppeteer） |
| 防御难度 | 低—改为 POST 就避住了 `<img>` | 需要 Token / SameSite / Origin 多重防御 |
| SameSite=Lax | 默认放行 | 默认拦截 |

auto-submit form 的三种写法:

| 写法 | 代码 | 适用 |
|---|---|---|
| JS `submit()` | `<script>document.forms[0].submit()</script>` | 最通用，本题首选 |
| `<iframe>` + onload | `<iframe name=t></iframe><form target=t>…</form>` | 隐蔽的后台提交 |
| 自动点击 submit | `<body onload=…>` | 旧时代调作 |

> Important  
> 为什么不能用 autofocus 动作触发: HTML5 `autofocus` 规范上不会触发 submit，只能用 JS 程序化 `submit()` / `.click()` 或 iframe 列等方式。

同源托管的威力——本题本质上提供了一个“Stored HTML 写入同源”的次生场景，当他会 XSS 一起看其实就是存储型 XSS。仅仅因为不返回自己 cookie，所以才跟 XSS 区开的。

| 问题 | 根因 | 解决 |
|---|---|---|
| 托管页打开就跳 但 bot 访问完 inbox 仍空 | bot 未上线 / Chromium 崩 | 等 45s+，并检查容器 supervisor 日志 bot 是否 RUNNING |

| 投递 URL 写成 `http://attacker.com/xxx` | bot 白名单只收 `/pages/` / `http://127.0.0.1/` | 写相对路径 |
| 用 requests 自己直接 POST /transfer | 用的是 student session，会扣 student 自己的余额 | 必须借助 bot 才能拿 admin 视角权限 |
| 转账成功但发送 inbox 空 | admin 之前已经给别人转过一次，本轮不再是首笔 | 重启容器或换 `to` 为其他名字 |

# CSRF 全谱新添加（POST 专章）

> Important

1. 经典 auto-submit form（本题）：`<form>+<script>submit()`
2. iframe 连接：`<iframe name=t></iframe><form target=t action=… method=post>…</form><script>form.submit()</script>` — 页面不跳转，如同无声暗 bot 眼皮底下
3. 文件上传 CSRF：`<form enctype="multipart/form-data">`，搭配 `程序化 FormData` 构造 JS 观察
4. JSON POST CSRF：`Content-Type: text/plain` + 伪装 JSON（避 preflight）
5. HTTP Method Override：一些框架 `?_method=PUT`，能用 POST 假装成 PUT / DELETE — 当心
6. 同站 Stored-HTML 托管 + bot（本题接同）：SameSite 无效，本质倾向 stored XSS

[代码块边界]python
ALLOW_ORIGIN = 'http://bank.example.com'
def check_origin():
    o = request.headers.get('Origin') or request.headers.get('Referer', '')
    return o.startswith(ALLOW_ORIGIN)
@app.before_request
def guard():
    if request.method == 'POST' and not check_origin(): abort(403)
