Q22 学生解题指引

这份指引面向初学者，只说明观察方法、推理链路和关键知识点；具体 payload、真实 flag/token 和敏感路径已用占位表达处理。

题目概览

先根据题目材料识别入口、认证状态、可控参数和预期保护目标，再通过本地靶机响应验证题型。

学习目标

  • 先做黑盒基线观察，不直接跳到答案。
  • 把每个请求的观察结果转化为下一步假设。
  • 用题面、源码提示或参考答案确认漏洞成因。
  • 理解修复方向，而不是只记一次性答案。

题面线索解读

本节根据原始题面归纳辅助指引，不展示题面原文。教师可用它引导学生把题面信息转化为可验证的观察步骤。

线索转化

  • 题面涉及 WAF/filter 时，按基线请求、被拦截模式、放行模式、推断规则、绕过思路、验证请求的顺序记录。
  • 题面涉及命令执行时，先确认参数是否拼接到系统命令，再判断是否有回显、过滤字符和可用外带通道。
  • 题面涉及文件读取或路径参数时，先确认根目录限制，再逐层尝试相对路径，判断是否能越过预期目录。

建议观察清单

  • 先发无害输入确认正常响应，再逐步加入关键字符或关键字，比较状态码、响应文本和执行效果。
  • 用基线参数确认正常功能，再用安全的分隔符测试是否改变输出；无回显场景优先找日志或 OOB 记录通道。
  • 比较正常文件、缺失文件和上级目录文件的响应差异，用错误信息缩小真实路径。
  • 每一步都应能回答：当前观察支持了哪个假设，下一步要排除哪个不确定点。

核心知识点

  • 过滤绕过的重点是理解规则边界，而不是直接记某个 payload。
  • 命令注入的核心是服务端把用户输入当作 shell 语法解释，修复应避免 shell 拼接并使用参数化 API。
  • 路径穿越题的关键是规范化路径和访问控制，不能只靠前端限制文件名。
  • 最终 flag 只是验证闭环；教学重点是让学生理解从题面线索到黑盒观察再到验证请求的推理过程。

黑盒观察与推理步骤

Q22 direct backend

以下步骤按 direct solver 的实际执行思路记录。每一步都对应脚本中的一个关键观察或请求动作，而不是只给最终结论。

Step 1：复现命令注入基线

操作

先按上一类 ping 功能建立正常请求，再用无害命令分隔测试注入是否仍存在。

预期观察

正常 ping 输出存在，但部分特殊字符或关键字会被拦截。

简要分析

本题重点通常是有过滤的命令注入，而不是发现注入点本身。

为什么要这样做

先复现基线，后续才能把失败归因于过滤规则。

当前步骤验证什么假设

验证注入点存在，同时确认加入了输入过滤。

Step 2：系统记录被拦截模式

操作

分别测试空格、路径分隔符、常见读取命令和分隔符变体，记录拦截提示或响应差异。

预期观察

某些字符或关键字会被拒绝，另一些 shell 语法仍可通过。

简要分析

黑名单通常只覆盖显眼写法，无法覆盖 shell 的等价表达。

为什么要这样做

把过滤规则写出来，学生才知道绕过是如何推断出来的。

当前步骤验证什么假设

验证过滤规则大致针对哪些字符或命令形式。

Step 3：寻找等价表达完成无害验证

操作

用 shell 变量、相对路径、替代命令或编码形式构造一个无害输出，确认绕过过滤后仍能执行。

预期观察

响应出现无害命令输出，而没有触发过滤提示。

简要分析

绕过不是魔法，而是利用 shell 中多种等价写法表达同一语义。

为什么要这样做

先绕过到无害输出，能证明规则缺口，而不急于读取最终文件。

当前步骤验证什么假设

验证是否存在被过滤规则遗漏的命令表达方式。

Step 4：在绕过链上定位和读取目标

操作

用已验证的等价表达列出当前目录或读取题目指定文件，必要时调整相对路径。

预期观察

响应返回目标内容或可定位目标文件的目录信息。

简要分析

当路径分隔和空白受限时，相对路径和 shell 变量常用于保持语义。

为什么要这样做

这一步把绕过规则和题目目标结合起来。

当前步骤验证什么假设

验证绕过后的命令是否具有足够文件读取能力。

Step 5：总结为什么黑名单不可靠

操作

记录建议：使用严格 IP 校验、参数数组调用系统命令或完全避免 shell，不能依赖关键字黑名单。

预期观察

学生能说明过滤绕过的原因和更稳妥的修复方式。

简要分析

本题核心是过滤规则覆盖不完整。

为什么要这样做

修复应转向白名单和安全 API，而不是继续补黑名单。

当前步骤验证什么假设

验证指引是否展示了“规则推断 -> 绕过验证 -> 结果读取”的顺序。

执行证据摘要

常见误区

  • 只复制参考答案，不先确认当前实例的接口和响应是否一致。
  • 忽略编码、Cookie、一次性 token、bot 轮询时间等执行条件。
  • 看到一次失败就换题型，而不是先记录响应并缩小假设。

修复建议

按题型回到根因修复：输入白名单、服务端鉴权、CSRF token、避免 shell 拼接、关闭危险 PHP wrapper、限制文件读取范围、使用强密钥和安全会话设计。

总结

本题应按“基线请求 -> 异常行为观察 -> 推断规则 -> 验证假设 -> 参考材料确认”的顺序完成。最终结果只作为验证闭环，不应替代前面的推理过程。
